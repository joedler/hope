# PRD_空空BOT開發手冊

最後更新：2026-06-05

## 產品定位

空空BOT 是「撐出空間教育關懷協會」用於講師授課登記、課程預排、預排核銷、行政單據與帳務補救的 LINE + LIFF + GAS + Google Sheets 工作台。

核心目標不是做大量功能入口，而是降低每月行政與講師操作錯誤，讓高風險作業都有預覽、確認、防呆與追溯紀錄。

## 使用者角色

- 講師：登記已完成授課、預排課程、核銷預排課程、查看自己的狀態。
- 行政人員：試算學費與鐘點、產生與寄送單據、處理收據/領據、帳務補救、權限與綁定排查。
- 家長：接收繳費單、收據與相關通知。
- 系統維護者：維護 GAS、LIFF、Google Sheets、LINE Messaging API 與資料規則。

## 核心設計原則

- 使用者入口以 LIFF 工作台為主，LINE 文字指令保留作為輔助與除錯。
- 所有會寫入、產生單據、寄送 Email、LINE push 的功能，一律採用「預覽 → 確認 → 執行」。
- 不直接覆蓋舊單據。修正舊月份錯誤時，使用帳務補救紀錄追溯處理。
- 資料應以 Google Sheets 為現階段主資料庫，但程式不可依賴脆弱欄位位置，應盡量用表頭與欄位名稱。
- 權限必須由後端再次檢查，前端隱藏入口不能視為安全控管。

## 講師端功能

- 我的狀態：顯示身份別、權限、本月已登記授課、待核銷預排、可登記學生、可登記課程。
- 授課登記：已上完課後，登錄學生、課程、日期、開始時間、結束時間。
- 預排課程：預先登記下個月或未來課程。
- 預排核銷：課程時間結束後，確認有上課或未上課。
- 多筆新增：以同一學生為主體，最多 5 筆，支援同學生同課程不同時間或同學生不同課程。

## 行政端功能架構

行政端應採用 Tab 工作台，不再以一堆大格子鋪滿畫面。

正式 Tab：

- 學費流程
- 收據流程
- 鐘點流程
- 帳務工具

## 行政端高風險規則

- 試算：學費試算、鐘點試算都先預覽，確認後才寫入。
- 繳費單：產生前先預覽，確認後才產生。後續支援寄送 Email、LINE push 給家長、查看已產生單據、重新產生、作廢、補發。
- 收據：產生前先預覽，確認後才產生。保留 Email 寄送，新增 LINE push 給家長。一般收據處理入會費、常年會費、捐款，至少需有預覽與寄送。
- 領據：產生前先預覽，確認後才產生。保留寄送功能，新增 LINE push 給講師。
- 帳務補救：處理漏登、短收、超收、補收、退費，必須關聯原繳費單，不直接改舊單。
- 補救明細/補收通知：針對帳務補救中的待處理補收產生補充通知，單號格式為 `ADJ_YYYY_MM_###`，不可修改原繳費單；正式月結金額仍併入處理月份學費試算。

## 行政預覽 API

- `adminPreview` 是只讀 API，只能回傳預覽資料，不可寫入、產生 PDF、寄送 Email 或 LINE push。
- `adminTask` 屬於執行型 API，不可由高風險入口直接呼叫。
- `adminConfirmSettlement` 僅開放學費試算與鐘點試算確認寫入；必須由預覽後的二次確認按鈕觸發，後端需重新試算並執行重複寫入防呆。
- `adminConfirmDocument` 用於已完成預覽後的單據 PDF 產生；目前先開放繳費單產生，不寄送 Email 或 LINE push。
- 前端行政預覽優先呼叫 `adminPreview`；若 API 失敗，需顯示明確錯誤，不再顯示開發期本機預覽。
- 預覽資料需包含：標題、月份、明細列、PDF 連結狀態、可選狀態與下一步按鈕文字；例行摘要與統計卡不得干擾主要核對。
- 財務預覽文字需採多行核對格式，避免將學生/講師、金額、日期、課程、狀態與原因擠在同一行。
- 預覽資料需由後端回傳 `canConfirm` 與 `confirmAction` 決定確認按鈕是否可按，前端不可只用功能名稱猜測可否執行。
- 資料來源統計只用於檢核資料是否存在，例如授課紀錄、預排紀錄、學費結算、鐘點結算、帳務補救筆數；不代表正式試算結果。
- 資料來源統計需回傳狀態，不可只回傳數字。狀態包含：`ok` 有資料、`empty` 本月無資料、`missing` 找不到分頁、`error` 讀取失敗。
- `0` 筆資料必須區分為「本月無資料」或「找不到分頁 / 讀取失敗」，避免行政人員誤以為資料正常。
- 學費試算預覽需先以只讀方式彙總待試算授課、預收下月預排、預收核對差異與帳務補救金額。
- 若指定月份已有學費結算資料，學費試算預覽應改為讀取既有學費結算摘要，不再重新計算授課、預排或帳務補救差異。
- 鐘點試算預覽需先以只讀方式彙總待試算授課、帳務補救中的補發鐘點、講師時數、應付總額、扣繳稅額、補充保費與實發金額。
- 若指定月份已有鐘點結算資料，鐘點試算預覽需改顯示既有結算摘要，確認寫入不可按；後端也必須阻擋同月份重複寫入。
- 已寫入的學費/鐘點試算再次預覽時，需以簡化狀態摘要顯示「已寫入」，不可顯示成像重新試算結果；學費仍需列出同月份待核銷預排並標示「待核銷，未寫入」。
- 若指定月份已有鐘點結算資料，預覽需提醒行政人員：回看舊領據或結算應使用「已產生單據 / 領據紀錄」，不可重新試算。
- 繳費單預覽需讀取既有學費結算表資料，顯示學生、單據編號、總金額、課程數與 PDF 產生狀態。
- 繳費單預覽不可重新試算學費、不可產生 PDF、不可寫入 PDF URL 或狀態欄、不可寄送 Email 或 LINE push。
- 繳費單確認產生只可在繳費單預覽後開放，產生 PDF 後回寫學費結算表 PDF URL 與狀態；不得同時寄送 Email 或 LINE push。
- 收據預覽需讀取學費結算表的學生、金額、單據編號、收款方式、收據類別與收款日期；收據 PDF 與寄送狀態必須讀取 `單據紀錄表` 中真正的收據紀錄，不可使用學費結算表中繳費單共用的 PDF/狀態欄位。
- 領據預覽需讀取鐘點結算表，顯示講師、應付總額、扣繳稅額、補充保費、實發金額、領據編號、PDF 與寄送狀態。
- 一般收據預覽需讀取一般收據紀錄，顯示姓名、類別、金額、編號與 PDF 狀態。
- 學費試算與正式寫入需納入處理月份的帳務補救補收/退費，讓試算金額、繳費通知、實際收款與收據能對齊。
- 學費試算確認寫入後，下一階段應進入繳費單預覽；鐘點試算確認寫入後，下一階段應進入領據預覽。
- 學費試算採課程列層級處理：待核銷預排顯示作核對，但在「寫入學費結算」模式下 checkbox 必須停用；確認寫入時只處理可寫入且已勾選的課程列。
- 試算頁是月結主流程中心：學費試算寫入後可直接在同一預覽區勾選學生產生繳費單；鐘點試算寫入後可直接在同一預覽區查看講師領據資料。
- 指定月份已有學費結算時，學費試算不得只顯示文字摘要；需帶入繳費單預覽資料，顯示學生、單號、金額、PDF 狀態與 `查看 PDF`，並同時列出待核銷預排作只讀核對。
- 預覽列呈現規則：所有資料以一致卡片呈現，顯示狀態、單號、PDF 與提醒；checkbox 是否可用由目前選定動作決定，不符合該動作條件者需在前端停用。
- 繳費單產生支援選取單一或多位學生，只處理已寫入且尚未產生 PDF 的學生；已產生、缺單號或缺金額的項目需顯示但 checkbox 停用。
- 預收課程若指定月份仍有未核銷預排，試算預覽需顯示為「待核銷」，不得直接列入退費差異；正式學費結算寫入前需阻擋並提示先完成核銷。
- 補救明細/補收通知預覽需讀取學費調整紀錄表中「調整類型 = 補收、狀態 = 待處理、尚未有補收單號」的資料。
- 補救明細/補收通知確認產生後，需產生 PDF，並回寫補收單號、補收 PDF、補收單狀態與狀態欄；原繳費單只保留為關聯，不可修改。
- 帳務補救採「來源事件獨立、帳務影響併入處理月份」：原月份與原單據不改，新補收/退費以調整紀錄與新單據追蹤；會計、收據、稅務依實際處理月份與實際收付款日期認列。
- 補救明細/補收通知不應自動寫入新的鐘點結算、收據或領據。若帳務補救是漏登已上課且可對應講師與課程，該筆需出現在處理月份的鐘點試算預覽中，標示為補發，經確認後才寫入鐘點結算。

## 時間欄位規格

- 顯示區一律為 24 小時制 `HH:MM`。
- 不允許顯示 `上午`、`下午`、`AM`、`PM` 或只顯示 `1:00`。
- 分鐘間隔固定為 `00 / 15 / 30 / 45`。
- 手機端可用原生時間選擇器；電腦端使用可控小面板。
- 詳細決策見 `doc/時間欄位設計決策紀錄.md`。

## 2026-06-11 領據產生流程更新

- 領據 PDF 產生已納入「預覽 → 勾選 → 確認 → 產生 PDF/回寫狀態」流程。
- 鐘點試算確認寫入後，行政人員可在領據預覽勾選講師並產生領據 PDF。
- 領據產生只回寫 PDF URL 與「待寄送」狀態，不會自動寄送 Email 或 LINE push。
- 領據寄送需走「寄送通知」預覽與二次確認；Email 與 LINE push 在同一卡片內各自判斷、各自勾選、各自回寫。

## 2026-06-11 收據產生流程更新

- 學費收據 PDF 產生已納入「預覽 → 勾選 → 確認 → 產生 PDF/回寫狀態」流程。
- 收據產生前必須已有單據編號、收款方式、收據類別、收款日期與收據金額；缺欄位者需顯示提醒，但在「產生收據」模式下 checkbox 停用。
- 收據產生只回寫 PDF URL 與「待寄送」狀態，不會自動寄送 Email 或 LINE push。
- 收據寄送需走「寄送通知」預覽與二次確認；Email 與 LINE push 在同一卡片內各自判斷、各自勾選、各自回寫。
## 2026-06-11 已產生單據總覽

- 行政端新增「已產生單據」只讀總覽。
- 總覽會彙整指定月份的繳費單、收據、領據、一般收據狀態。
- 此功能只用於回看與核對，不會寫入、不會寄送、不會作廢、不會重新產生。
- 作廢、重新產生、補發屬於高風險動作，後續必須另行設計「預覽 → 確認 → 執行」流程後才可開放。

## 2026-06-11 收據與領據 Email 寄送

- 行政端新增「寄送收據 Email」與「寄送領據 Email」預覽確認流程。
- 寄送前需先預覽待寄送清單，確認 PDF、Email、狀態皆正確後才可勾選寄出。
- 只有狀態為「待寄送」、已有 PDF、且有 Email 的項目可勾選。
- 寄送完成後回寫寄送狀態；失敗時回寫失敗原因。
- 繳費單 Email 因目前與收據共用部分欄位，需另行拆清狀態欄後再開放。

## 2026-06-11 收據收款資料套用與試算流程整合

- 收據預覽新增收款資料套用：收款方式、收據類別、收款日期。
- 套用後會寫入指定月份尚未產生收據 PDF 的學費結算資料。
- 收據產生仍需重新預覽確認，符合欄位完整、尚未產生 PDF、金額與單據編號存在者才可勾選。
- 行政端流程分頁保留可執行的作業入口；`已產生單據` 不再放在流程分頁中，改由帳務工具的單據總覽供需要時查核。
- 獨立分頁暫時保留，主操作方向逐步收斂到試算分頁下方。

## 2026-06-11 單據紀錄表與三流程定案

- 新增 `單據紀錄表` 作為繳費單、收據、領據、一般收據、補收通知的 PDF、Email、LINE push、作廢、補發狀態來源。
- 學費流程：學費試算 → 確認寫入 → 產生繳費單 → 寄送通知（Email + LINE）。
- 收據流程：收款資料 → 確認收款 → 產生收據 → 寄送通知（Email + LINE）。
- 鐘點流程：鐘點試算 → 確認寫入 → 產生領據 → 寄送通知（Email + LINE）。

## 2026-06-11 單據紀錄表整合

- 「已產生單據」總覽改以 `單據紀錄表` 為主要來源，集中顯示繳費單、補收通知、收據、領據、一般收據的 PDF、Email、LINE push 與作廢狀態。
- 收據與領據 Email 寄送結果需同步回寫 `單據紀錄表`，包含已寄送、缺 Email、PDF 連結錯誤與寄送失敗原因。
- 原本的學費結算表、鐘點結算表仍保留結算結果與相容欄位，但單據後續管理、補發、作廢、LINE push 應以 `單據紀錄表` 為準。
- `學費結算表` 與 `鐘點結算表` 保留結算結果，不再作為所有單據狀態的唯一來源。

## 2026-06-11 單據 Email 預覽分流

- 繳費單產生後，`單據紀錄表` 的 Email 狀態應為 `待寄送`，因為正式學費流程下一步是寄送通知（Email + LINE）。
- 收據 Email 預覽只能讀 `單據紀錄表` 中 `單據類型=收據` 的資料，不可讀到繳費單狀態。
- 領據 Email 預覽只能讀 `單據紀錄表` 中 `單據類型=領據` 的資料。
- 繳費單 Email 先開放只讀預覽，正式寄送仍需後續接上預覽後二次確認流程。

## 2026-06-11 行政端入口收斂

- 學費流程顯示：學費試算；寫入學費結算、產生繳費單、寄送通知放在資料卡片上方。
- 收據流程顯示：收款資料、一般收據；套用收款資料、產生收據、寄送通知放在資料卡片上方。
- 鐘點流程顯示：鐘點試算；寫入鐘點結算、產生領據、寄送通知放在資料卡片上方。
- 原「試算後續流程」與獨立繳費單 TAB 移除，避免同一功能在多處重複出現。

## 2026-06-12 行政流程狀態提示

- 行政流程入口只保留主要步驟名稱，不再用「可執行 / 只讀 / 尚未開放」作為入口卡片說明。
- 預覽明細卡需顯示資料狀態、提醒、單號與 PDF；行政人員先勾選資料，再按下寫入、產生、Email 或 LINE。
- 動作按鈕先切換處理模式，不直接執行；切換後只有符合該動作條件的卡片 checkbox 可用，確認按鈕只在已勾選可處理項目時啟用。
- 尚未定義資料來源的功能不可呈現為正式動作按鈕。

## 2026-06-12 繳費單 Email 正式寄送

- 繳費單 Email 從只讀預覽改為可勾選寄送。
- 寄送流程：繳費單 Email 預覽 → 勾選學生 → 二次確認 → Gmail 寄送附件 → 回寫 `單據紀錄表`。
- 寄送成功需更新 `單據紀錄表` 的 Email狀態為 `已寄送`，並寫入寄送時間與備註。
- 缺 Email、PDF 連結錯誤或 Gmail 例外時，需回寫失敗狀態與原因。
- LINE push 仍未開放，需另行確認家長 LINE 綁定與推播文案。

## 2026-06-12 單據 Email 資料來源補強

- 若既有資料尚未同步到 `單據紀錄表`，但原結算表已有明確對應的 PDF 與待寄送狀態，預覽需 fallback 顯示該筆資料。
- 繳費單 fallback 來源為 `學費結算表` 既有繳費單 PDF；領據 fallback 來源為 `鐘點結算表` 既有領據 PDF。收據 Email 不可 fallback 讀學費結算表 PDF 欄位，因該欄位可能是繳費單 PDF。
- 使用 fallback 資料寄送 Email 時，系統必須先回補 `單據紀錄表`，再更新 Email 寄送狀態。
- 此規則用來處理舊流程已產生 PDF、但新流程尚未建立單據中心紀錄的資料落差。
- fallback 顯示時必須保留原結算表狀態：`待寄送` 可勾選寄送，`已寄送` 只列入已寄送統計且不可重複勾選。

- 每類單據需顯示：筆數、PDF 已產生數、Email 待寄送/已寄送/失敗、LINE 未推播/已推播、金額摘要。

## 2026-06-12 單據版本管理規則

- 「已產生單據」已改名並收斂為帳務工具內的「單據總覽」；作廢、補發、重新產生可在單據卡片上方選擇操作。
- 作廢、補發、重新產生都是高風險單據動作，必須另走「預覽 → 確認 → 執行」流程。
- 作廢：保留 `單據紀錄表` 原列、PDF 連結、Email/LINE 狀態與金額，不刪除資料；只更新作廢狀態、備註與操作人。作廢後不可再寄送 Email 或 LINE push。
- 補發：同一份 PDF 重新寄送給原對象，內容、金額、單據編號不變。若金額、收款資料、課程內容或收件人需更正，不可用補發，應走作廢與新版流程。
- 重新產生：尚未寄送的單據可重新產生 PDF 並更新紀錄；已寄送或已 LINE push 的單據不可靜默覆蓋，需先作廢原單，再建立新版或更正版。
- 繳費單、收據、領據、一般收據與補收通知的後續狀態，以 `單據紀錄表` 為主要依據。
## 2026-06-12 LINE push 前置檢查

- 繳費單與收據的 LINE push 對象為家長/學生，來源為 `學生基本資料表` 中的 LINE User ID 欄位。
- 領據的 LINE push 對象為講師本人，來源為 `講師名單` B 欄 LINE User ID。

- 正式 LINE push 後續仍必須另走「預覽 → 確認 → 執行」，不得在總覽頁直接推播。
## 2026-06-12 LINE push 正式流程

- LINE push 第一版採純文字訊息，不使用 Flex/message 卡片。
- 推播內容包含月份、姓名、單據類型、單據編號、金額與 PDF 連結。
- 繳費單與收據 LINE push 讀取 `學生基本資料表` D 欄的家長 LINE User ID。
- 領據 LINE push 讀取 `講師名單` B 欄的講師 LINE User ID。
- 行政端新增三個功能：`寄送繳費單 LINE`、`寄送收據 LINE`、`寄送領據 LINE`。
- 所有 LINE push 均需先預覽、勾選對象、二次確認後才執行，成功或失敗需回寫 `單據紀錄表` 的 LINE 狀態、寄送時間與備註。
- 過渡期若既有領據 PDF 尚未同步到 `單據紀錄表`，領據 LINE push 預覽可從 `鐘點結算表` 補讀；實際推播前需先回補 `單據紀錄表`。
- 行政端工具區提供 `作廢單據`、`補發單據`、`重新產生單據` 入口；三者皆需先預覽、勾選、二次確認後才可執行。
- `作廢單據` 已開放正式流程：預覽有效單據、勾選、二次確認、輸入作廢原因後，僅更新 `單據紀錄表` O/P/Q 欄，不刪除 PDF 或資料列。
- `補發單據` 已開放正式流程：預覽有效且有 PDF 的繳費單、收據、領據，勾選後可選 Email、LINE 或 both 補發同一份 PDF；不改金額、內容或單據編號。
- `重新產生單據` 已開放正式流程：只允許未 Email 寄送、未 LINE 推播、未作廢且已有 PDF 的繳費單、收據、領據。執行時需填寫原因，系統會將舊 PDF 移至垃圾桶、重新產生 PDF、回寫 `單據紀錄表` K/P/Q 欄；已對外寄送或推播的單據必須先作廢，再建立新版。

## 2026-06-13 行政流程 Eligibility Matrix

- 學費流程
  - 寫入學費結算：只允許待寫入且可結算的課程列；待核銷預排、已寫入、缺資料不可勾選。

- 收據流程
  - 套用收款資料：只允許已有學費結算、尚未產生收據 PDF、單號與金額完整的學生列。
  - 產生收據：只允許已有收款方式、收據類別、收款日期、尚未產生收據 PDF 的學生列；不可把繳費單 PDF 當成收據 PDF。
  - 寄送通知：只允許已有收據 PDF 且 Email 待寄或 LINE 未推播的資料；缺收據 PDF、只有繳費單 PDF、已寄送或已推播不可勾選對應通道。
- 鐘點流程
  - 寫入鐘點結算：只允許待寫入鐘點資料與符合條件的帳務補救補發；已寫入、重複、缺講師資料不可勾選。

- 帳務工具
  - 帳務補救：寫入前以處理月份、學生、課程、日期、時間、類型防重。
  - 作廢：只允許有效且未作廢單據。
  - 補發：只允許有 PDF 且未作廢單據。
  - 重新產生：只允許未 Email 寄送、未 LINE 推播、未作廢且已有 PDF 的單據。
