# 開發環境基礎建設懶人包

## 專案定位

本專案採用：

- Google Sheets 作為資料庫
- Google Apps Script 作為 API 後端
- LINE / LIFF 作為使用者入口
- GitHub Pages 作為前端靜態網站
- Git / GitHub 作為版本控管
- TypeScript + clasp 作為 GAS 本地開發與部署流程

目標是讓單案能快速交付，未來也能複製到不同學校、旅行團或客戶。

## 全域環境與專案環境

基礎建設要分成「全域工具」與「專案設定」。

全域工具是安裝在電腦或開發者帳號上的能力，通常所有專案共用。

建議全域工具：

- Git
- Node.js
- npm
- GitHub 帳號
- Google 帳號
- LINE Developers 帳號
- clasp 登入授權

專案環境是跟著每個客戶或每個 repo 走的設定，不應混用。

本專案內保存：

- `package.json`
- `package-lock.json`
- `gas/.clasp.json`
- `gas/tsconfig.json`
- `eslint.config.mjs`
- `gas/src/Code.ts`
- `web/`，前端建立後使用
- 文件與資料轉換腳本

Apps Script 專案屬性保存每案不同的設定：

- `SPREADSHEET_ID`
- `PEOPLE_SHEET`
- `SETTINGS_SHEET`
- `API_KEY`
- `LIFF_ID`，前端需要時加入
- `WEB_APP_URL`，前端需要時加入

Google Sheets `settings` 工作表保存可由客戶調整的畫面設定：

- `activity_name`
- `table_meal_label`
- `room_number_status`

這類設定適合給客戶直接在 Google Sheets 修改，不需要重新部署前端。

不要放進 Git 的內容：

- `node_modules/`
- `gas/build/`
- 私密 token
- 客戶敏感個資
- 本機暫存檔

簡單判斷：

- 換一台電腦也會共用的是全域工具。
- 換一個客戶就會不同的是專案設定。
- 不能公開或不能交給所有人的，放到專案屬性或 GitHub Secrets，不寫進程式碼。

## 資料歸屬原則

- 客戶 Google Drive：正式名單、行程、領隊、房號、正式 Google Sheets。
- 我方 GitHub / 本機專案：程式碼、轉換工具、模板、文件。
- Apps Script：可建在客戶端或我方端，依權限與維護模式決定。

建議：

- 一次性專案：每案獨立 Google Sheets + 每案獨立 GAS。
- 產品化後：共用核心程式 + 多客戶設定。

## 重要 ID 管理

不要把重要 ID 寫死在主流程程式中。

GAS 使用 Apps Script 專案屬性保存：

- `SPREADSHEET_ID`
- `PEOPLE_SHEET`
- `SETTINGS_SHEET`
- `API_KEY`
- 未來可加 `LIFF_ID`
- 未來可加 `LINE_CHANNEL_ID`
- 未來可加 `LINE_CHANNEL_SECRET`

第一次建立 GAS 後，執行：

```text
setupProjectProperties
```

再到 Apps Script：

```text
專案設定 -> 指令碼屬性
```

確認或修改正式值。

## 專案資料表標準

正式 Google Sheets 建議包含：

- `people`：查詢主資料
- `settings`：活動設定
- `itinerary`：行程表
- `contacts`：領隊與聯絡窗口

`people` 最低必要欄位：

- `person_id`
- `類別`
- `姓名`
- `班級或職稱`
- `學號`
- `手機末三碼`
- `LINE使用者ID`
- `綁定時間`
- `車次`
- `桌號`
- `房號`
- `房間編組`
- `同房人員`
- `素食註記`

## 本地開發流程

修改 GAS：

```text
gas/src/Code.ts
```

不要直接修改：

```text
gas/build/Code.js
Apps Script 線上編輯器程式碼
```

常用指令：

```powershell
npm.cmd run gas:lint
npm.cmd run gas:build
npm.cmd run gas:check
npm.cmd run gas:push
```

標準流程：

1. 修改 `gas/src/Code.ts`
2. 執行 `npm.cmd run gas:check`
3. 執行 `npm.cmd run gas:push`
4. 到 Apps Script 後台更新 Web App 部署版本
5. 測試 `?action=health`

## Git / GitHub 建議

建議分支：

- `main`：正式穩定版
- `develop`：開發版
- `feature/*`：新功能

建議 commit 類型：

- `feat:` 新功能
- `fix:` 修正
- `docs:` 文件
- `chore:` 環境或設定
- `data:` 資料轉換或名單修正

建議 GitHub Actions：

- push 時跑 `npm.cmd run gas:check`
- 前端完成後自動部署 GitHub Pages
- 未來可加自動 `clasp push`

## LINE / LIFF 建議

LINE Bot 定位：

- 查詢工具
- 自動說明
- 非即時客服

第一版功能：

- 第一次驗證並綁定 LINE ID
- 已綁定使用者自動查詢
- 解除綁定
- 顯示車次、桌號、房號、房間編組、同房人員、素食註記

使用者說明需明確寫：

```text
本系統提供畢旅資訊查詢，不是即時客服。
臨時狀況、資料異動或個人問題，請聯絡班導、隨隊老師或旅行社窗口。
```

## 前端 / GitHub Pages 建議

前端建議獨立放：

```text
web/
```

前端可先採輕量架構：

- HTML
- Tailwind CSS CDN
- Alpine.js
- LIFF SDK
- GitHub Pages

適用情境：

- 管理後台
- 查詢頁面
- 小型設定頁
- 不需要大型 SPA 框架的專案

建議檔案結構：

```text
web/
  index.html
  admin.html
  config.js
  assets/
  README.md
```

前端設定建議放：

```text
web/config.js
```

最低設定：

- GAS Web App URL
- LIFF ID
- 活動名稱

Tailwind CSS CDN 建議：

- 適合 MVP、單案、管理後台快速交付。
- 不需要建置流程即可部署到 GitHub Pages。
- 若未來 UI 複雜或要最佳化效能，再改成 Tailwind CLI / Vite。

Alpine.js 建議：

- 適合表單、狀態切換、簡易互動。
- 適合與 GAS API 搭配。
- 不適合做大型複雜 SPA；若功能變多再評估 Vue / React。

GitHub Pages 建議：

- `main` 分支部署正式版。
- 可用 GitHub Actions 自動部署。
- 前端設定可分 `config.dev.js`、`config.prod.js`，但正式頁只載入 production 設定。

前端 API 呼叫原則：

- 只呼叫 GAS Web App URL。
- 不直接讀 Google Sheets。
- 不暴露 LINE Channel Secret。
- 不保存管理者高權限 token。
- 若有管理後台，需加簡單登入或管理 token。

不要在前端保存：

- LINE Channel Secret
- 客戶完整個資
- 可管理資料的私密 token

前端最小功能：

- 首次驗證畫面
- 已綁定查詢畫面
- 解除綁定按鈕
- 錯誤訊息顯示
- 非即時客服說明

管理後台最小功能：

- 查看 API health
- 查詢綁定狀態
- 解除指定 LINE 綁定
- 查看活動設定
- 之後可加行程與領隊資料維護

## 安全與隱私

前端不顯示：

- 身分證
- 生日
- 完整手機
- Google Sheet 原始連結

系統表只保留查詢必要資料：

- 學生：學號
- 老師：手機末三碼
- 綁定後：LINE 使用者 ID

建議 API 後續加入：

- `API_KEY` 驗證
- 操作紀錄
- 綁定失敗次數限制
- 管理者解除綁定流程

## 新客戶複製 Checklist

1. 複製 Google Sheets 範本。
2. 匯入新客戶名單。
3. 產出或整理 `people/settings/itinerary/contacts`。
4. 建立 Apps Script 專案。
5. 設定 `.clasp.json` 的 `scriptId`。
6. 執行 `npm.cmd run gas:push`。
7. 執行 `setupProjectProperties`。
8. 設定正式 `SPREADSHEET_ID`。
9. 部署 Web App。
10. 測試 `health`、`me`、`verifyAndBind`。
11. 建立 LIFF / GitHub Pages 前端。
12. 產出 QR Code。
13. 客戶測試。
14. 上線前備份資料。
