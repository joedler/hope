# 測試清單與驗收規格

最後更新：2026-06-05

## 通用驗收原則

- 所有會寫入、產生單據、寄送 Email、LINE push 的功能，都必須先顯示預覽。
- 未確認前不可寫入。
- 權限不足者不可看到或執行行政功能。
- 前端擋住之外，後端也必須檢查權限。
- 錯誤情境必須有明確提示，不可只有無反應。

## LIFF 講師端

### 我的狀態

- 顯示身份別。
- 顯示權限。
- 顯示本月已登記授課。
- 顯示待核銷預排。
- 顯示可登記學生。
- 顯示可登記課程。
- 點選數字後顯示對應資料。

### 授課登記

正常流程：

- 選學生後，只出現該學生課程。
- 日期、時間、課程完整後可送出。
- 成功後寫入授課紀錄。

錯誤流程：

- 空欄位不可送出，需顯示提示。
- 未來時間不可登記為已授課。
- 結束時間早於開始時間不可送出。
- 課程時數過長不可送出。
- 同批新增時間重疊不可送出。
- 同老師同時段重疊不可寫入。
- 同學生同時段重疊不可寫入。

### 預排課程

正常流程：

- 可登記未來課程。
- 成功後寫入預排紀錄。
- 成功後出現在預排核銷清單。

錯誤流程：

- 過去時間不可預排。
- 完全重複預排不可重複寫入。
- 同老師、同學生、同時段重疊需提示。

### 預排核銷

正常流程：

- 未來預排可顯示在清單。
- 課程時間結束後可核銷。
- 有上課後轉入或更新授課紀錄。
- 未上課後取消預排並進入帳務規則。

錯誤流程：

- 未來時間不可核銷。
- 同一筆預排核銷後，再點一次應被擋下。
- 無權限或非本人資料不可核銷。

## 行政端

### 試算

- 學費試算先顯示預覽。
- 鐘點試算先顯示預覽。
- 未確認不可寫入。
- 確認後才寫入試算結果。
- 重複試算不可產生重複錯誤資料。
- 在正式預覽 API 接上前，試算入口不可直接呼叫寫入型後端任務。
- 前端預覽卡需顯示目前狀態與停用的下一步按鈕。
- 切換行政 Tab 後，上一個預覽卡應清除。
- `adminPreview` API 必須是只讀 API，不可寫入任何試算或單據資料。
- `adminPreview` API 權限不足時需回傳錯誤，不可提供行政預覽。
- `adminPreview` API 可回傳資料來源統計，但不得產生正式結算資料。
- 前端預覽卡需顯示資料來源統計區塊。
- 財務預覽文字需保留換行並分段顯示，至少區分姓名、金額、課程/明細、單號、狀態與提醒。
- 資料來源統計需區分 `ok`、`empty`、`missing`、`error` 四種狀態。
- 當指定月份無資料時，前端需顯示「本月無資料」，不可只顯示 `0`。
- 當 Google Sheets 缺少分頁時，前端需顯示「找不到分頁」，不可被誤認為正常 `0` 筆。
- 當讀取資料失敗時，前端需顯示「讀取失敗」，不可繼續進入寫入或寄送流程。
- 學費試算預覽需顯示學生/課程層級的只讀彙總，不可寫入學費結算表。
- 學費試算預覽需納入預收下月預排、預收核對差異、後收實上課程與帳務補救金額。
- 學費試算預覽若有未核銷預排，需顯示為待核銷並暫不計入退費；待核銷課程列不可勾選，但其他可結算課程列仍可勾選寫入。
- 學費試算確認寫入後端需只接受已勾選且可結算的學生/課程；待核銷預排即使被前端偽造送出也不可寫入。
- 當指定月份已有學費結算資料時，預覽需顯示「回看舊單應使用已產生單據，不要重新試算」類型提醒。
- 學費試算確認寫入需經二次確認，後端需重新試算並寫入學費結算表。
- 學費試算確認寫入需阻擋同月份同學生同課程重複寫入。
- 學費試算確認寫入後，授課紀錄與預排紀錄只可回溯標記已寫入的學生/課程，不可整月份全部標記。
- 學費試算確認寫入後，收費單編號必須使用 `R_YYYY_MM_###`，例如 `R_2026_06_001`，不可使用民國年短碼如 `R1506_001`。
- 學費結算表 A 欄應為結算月份，B 欄應為學生姓名，J 欄應為收費單/單據編號。
- 學費試算確認寫入完成後，繳費單預覽需讀到新增的學費結算資料。
- 鐘點試算預覽需顯示講師層級的只讀彙總，不可寫入鐘點結算表。
- 鐘點試算預覽需納入待試算授課、講師時數、應付總額、扣繳稅額、補充保費與實發金額。
- 當指定月份已有鐘點結算資料時，預覽需顯示「回看舊領據或結算應使用已產生單據 / 領據紀錄，不要重新試算」類型提醒。
- 鐘點試算確認寫入需經二次確認，後端需重新試算並寫入鐘點結算表。
- 鐘點試算確認寫入需阻擋同月份同講師重複寫入。
- 指定月份已有鐘點結算資料時，鐘點試算預覽需顯示既有結算摘要，確認寫入按鈕不可按；後端直接呼叫確認寫入也必須回錯誤，不可新增列。
- 鐘點試算確認寫入後，領據編號必須使用 `A_YYYY_MM_###`，例如 `A_2026_06_001`，不可使用 `P1506_001`。
- 鐘點結算表 A 欄應為結算月份，B 欄應為講師姓名，C 欄為課程/學生，D 欄為課程日期/時間(金額)，E 欄為時數/費率，F 欄為單科試算，G 欄為應付小計，H 欄為單據編號；補發註記應寫入 J 欄備註。
- 鐘點試算確認寫入完成後，領據預覽需讀到新增的鐘點結算資料。

### 繳費單

- 產生前顯示預覽。
- 預覽需包含學生、課程、月份、預收/後收、補收/退費、總金額。
- 確認後才產生繳費單。
- 寄送 Email 前需顯示收件人與內容摘要。
- LINE push 前需確認家長 LINE 綁定。
- 已產生單據可查看。
- 後續測試重新產生、作廢、補發。
- 繳費單預覽需讀取學費結算表，顯示學生、單據編號、總金額、課程數與 PDF 產生狀態。
- 繳費單預覽不可重新試算學費，不可產生 PDF，不可寫入 PDF URL 或狀態欄。
- 繳費單預覽不可寄送 Email 或 LINE push。
- 繳費單預覽需由後端回傳 `canConfirm` 與 `confirmAction` 控制確認按鈕，前端不可只用功能名稱開放高風險按鈕。
- 繳費單預覽需回傳可勾選列；已產生、缺單據編號、缺金額者不可勾選。
- 繳費單確認產生只可處理勾選學生；未勾選者不可產生。
- 繳費單確認產生需經二次確認，產生 PDF 後回寫學費結算表 PDF URL 與狀態。
- 繳費單確認產生不可寄送 Email 或 LINE push。
- 若該月份繳費單已全部產生，確認按鈕需停用，不可重複產生 PDF。

### 收據

- 產生前顯示預覽。
- 收據預覽需讀取學費結算表的學生、金額、單據編號、收款方式、收據類別與收款日期；收據 PDF 與寄送狀態必須讀取 `單據紀錄表` 中真正的收據紀錄。
- 確認後才產生收據。
- 寄送 Email 前需確認收件人。
- LINE push 前需確認家長 LINE 綁定。
- 一般收據需測入會費、常年會費、捐款。
- 一般收據預覽需讀取一般收據紀錄，顯示姓名、類別、金額、編號與 PDF 狀態。
- 在正式預覽 API 接上前，收據與一般收據入口只可顯示預覽或暫停提示，不可直接產生或寄送。

### 領據

- 產生前顯示預覽。
- 預覽需包含講師、月份、授課紀錄、鐘點費、補充項目。
- 領據預覽需讀取鐘點結算表，顯示講師、應付總額、扣繳稅額、補充保費、實發金額、領據編號、PDF 與寄送狀態。
- 確認後才產生領據。
- LINE push 前需確認講師 LINE 綁定。
- 領據 PDF 產生已接上正式預覽確認流程：可勾選尚未產生 PDF 且有領據編號的講師，確認後產生 PDF 並回寫「待寄送」。
- 領據產生不可自動寄送 Email 或 LINE push；寄送需另走預覽與二次確認流程。
- 學費收據 PDF 產生已接上正式預覽確認流程：可勾選尚未產生 PDF，且已有單據編號、收款方式、收據類別、收款日期與金額的學生，確認後產生 PDF 並回寫「待寄送」。
- 收據產生不可自動寄送 Email 或 LINE push；寄送需另走預覽與二次確認流程。

### 帳務補救

- 可選補收或退費。
- 可選原錯誤月份與處理月份。
- 選學生後，只出現該學生課程。
- 選課程後，自動帶出單價。
- 選時間後，自動計算時數。
- 可選關聯原單號。
- 建立前需顯示補救預覽。
- 確認後才寫入補救紀錄。
- 不可直接改舊繳費單。
- 學費試算預覽與正式寫入需納入處理月份的帳務補救補收/退費。
- 學費試算預覽中的帳務補救需明確標示補收/退費、原月份、上課日期、開始/結束時間、課程名稱、時數、單價、金額、原單號、狀態與原因。
- 補救明細/補收通知預覽只可列出補收、待處理、尚未有補收單號的資料。
- 補救明細/補收通知確認產生後需產生 PDF，單號格式為 `ADJ_YYYY_MM_###`。
- 補救明細/補收通知產生後需回寫補收單號、補收 PDF、補收單狀態，並將原補救紀錄狀態改為已產生補收單。
- 補救明細/補收通知產生不可修改原繳費單、原單號或原 PDF。
- 非行政不可預覽或產生補救明細/補收通知。
- 補救明細/補收通知產生不可自動寫入新的鐘點結算、收據或領據。
- 帳務補救若為補收且有時數、可對應課程設定中的講師，處理月份的鐘點試算預覽需顯示補發鐘點與金額。
- 鐘點試算預覽中的帳務補救補發需明確標示原月份、原上課日期、開始/結束時間、學生、課程名稱、時數與補發金額。
- 鐘點試算確認寫入時，補發鐘點需寫入處理月份的鐘點結算表，不回頭修改原錯誤月份。
- 已有學費結算資料的月份，學費試算預覽需顯示既有結算摘要，不可重新計算授課、預排或帳務補救差異。
- 已有學費結算資料的月份，學費試算預覽文字需簡化並明確顯示「已寫入」；同月份未核銷預排仍需列出，狀態為「待核銷，未寫入」。
- 已有鐘點結算資料的月份，鐘點試算預覽文字需簡化並明確顯示「已寫入」，不得顯示成可重新試算寫入。
- 歷史補救已退款或已結清的月份，回看時不可再顯示舊差異造成的欠款或負數應收。
- GAS manifest 必須包含 `https://www.googleapis.com/auth/documents`，否則產生補收 PDF 時會因 `DocumentApp.openById` 權限不足失敗。

## 時間欄位

- 手機顯示區為 `HH:MM`。
- 電腦顯示區為 `HH:MM`。
- 電腦自訂面板小時為 `06-18`。
- 分鐘為 `00 / 15 / 30 / 45`。
- 不可顯示 `上午`、`下午`、`AM`、`PM` 作為表單顯示值。

## 回歸測試

每次修改 LIFF 後至少測：

- 講師身份登入。
- 行政身份登入。
- 非行政無法進入行政作業。
- 授課登記一筆成功。
- 預排課程一筆成功。
- 未來預排不可核銷。
- 行政高風險功能不可直接寫入。
- 行政端讀取預覽、產生補救明細、建立帳務補救等操作需顯示等待/執行中提示，避免使用者誤以為沒有點選成功。
- 「解除 LINE 帳號綁定」只應出現在我的狀態底部，其他操作頁不可重複顯示。
- 2026-06-11 補充驗收：帳務補救預覽在手機版不可出現空白外框；補救明細文字需保留換行並分段顯示，不得擠成一整行；等待 API 回應時需有可見的讀取提示。
## 已產生單據總覽測試

- 行政人員可在行政作業點選「已產生單據」並看到指定月份總覽。
- 總覽需以 `單據紀錄表` 為主要來源，包含繳費單、補收通知、收據、領據、一般收據的筆數、PDF 狀態、Email 狀態、LINE push 狀態與金額摘要。
- 若指定月份 `單據紀錄表` 無資料，需明確顯示目前沒有可查看的單據資料，不可誤導為重新試算。
- 此頁為只讀查看，不可出現可執行的確認寫入、寄送、作廢、重新產生或補發按鈕。
- 非行政人員呼叫 `adminPreview` 的「已產生單據」功能時，必須回覆權限不足。
- 若某一類單據資料表不存在或讀取失敗，總覽需顯示該類讀取失敗，不可讓整個行政頁崩潰。

## 收據與領據 Email 寄送測試

- 寄送收據 Email 與寄送領據 Email 必須先顯示預覽，不可直接寄送。
- 只有「待寄送」、已有 PDF、且有 Email 的項目可以勾選。
- 缺 PDF、缺 Email、已寄送、或狀態不是待寄送的項目不可勾選，且需顯示提醒。
- Email 寄送成功後，原結算表與 `單據紀錄表` 皆需更新為已寄送。
- Email 寄送失敗時，`單據紀錄表` 需保留失敗原因，包含缺 Email、PDF 連結錯誤或 Gmail 例外訊息。
- 繳費單 Email 預覽需可列出 `單據紀錄表` 中 `單據類型=繳費單` 且待寄送的資料。
- 繳費單 Email 可寄送項目需有 PDF、有效 Email、狀態為待寄送。
- 繳費單 Email 寄送成功後，`單據紀錄表` Email狀態需更新為已寄送，並寫入寄送時間與備註。
- 收據 Email 預覽只能列出 `單據紀錄表` 中 `單據類型=收據` 的資料，不可列出繳費單。
- 領據 Email 預覽只能列出 `單據紀錄表` 中 `單據類型=領據` 的資料。
- 繳費單產生後，`單據紀錄表` 的 Email 狀態需為 `待寄送`；既有舊資料若為 `未寄送`，前端總覽需以待寄送語意呈現。
- 確認寄送後需回寫「已寄送」；失敗時需回寫失敗原因。
- 未勾選任何可寄送項目時，不可執行寄送。
- 非行政人員不可呼叫寄送預覽或確認寄送。

## 收據收款資料套用測試

- 收據預覽需顯示收款資料套用區：收款方式、收據類別、收款日期。
- 未填完整三欄不可送出。
- 套用後需寫入學費結算表的收款方式、收據類別、收款日期欄位。
- 已產生收據 PDF 的資料不可被批次套用覆蓋。
- 套用後重新讀取收據預覽，符合條件者應可勾選產生收據。
- 非行政人員不可呼叫收款資料套用 API。

## 行政流程入口測試

- 行政端 TAB 需顯示：學費流程、收據流程、鐘點流程、帳務補救、工具。
- 學費流程需包含：學費試算、產生繳費單、寄送繳費單、LINE push。
- 收據流程需包含：收款資料、產生收據、寄送收據、LINE push、一般收據。
- 鐘點流程需包含：鐘點試算、產生領據、寄送領據、LINE push。
- 行政流程分頁不得再出現 `已產生單據` 按鈕；單據查核與版本管理應走工具區的作廢、補發、重新產生流程，或保留後端只讀總覽供必要時查核。
- 不可再顯示舊的「試算後續流程」或獨立「繳費單」TAB，避免同功能多處重複。
- 流程卡片需標示「可執行」或「只讀」，不可讓只讀或尚未開放功能看起來可直接執行。
- 預覽區狀態提示需依情境顯示：可勾選確認、尚未勾選、目前無可執行項目、或只讀核對。
- 入口只負責導向預覽，不可繞過預覽確認直接寫入、產生或寄送。

## 單據 Email fallback 測試

- 若 `單據紀錄表` 有繳費單、收據、領據紀錄，Email 預覽需以該表為主要來源。
- 若 `單據紀錄表` 缺繳費單紀錄，但 `學費結算表` 已有繳費單 PDF，寄送繳費單 Email 預覽需顯示該學生。
- 若 `單據紀錄表` 缺收據紀錄，即使 `學費結算表` 有繳費單 PDF，寄送收據 Email 預覽也不可顯示該學生；需先完成產生收據 PDF。
- 若 `單據紀錄表` 缺領據紀錄，但 `鐘點結算表` 已有領據 PDF 且狀態為待寄送，寄送領據 Email 預覽需顯示該講師。
- 使用 fallback 顯示的資料執行 Email 寄送後，需回補 `單據紀錄表`，並更新 Email 狀態、寄送時間與備註。
- fallback 資料若原結算表狀態為已寄送，預覽需列入已寄送統計，但不可勾選重複寄送。
- Email 確認寄送端必須使用與預覽端相同的狀態正規化規則，避免預覽可勾選但送出時找不到待寄送資料。

## 已產生單據總覽顯示測試

- 已產生單據總覽為只讀核對頁，不可顯示可勾選執行。
- 總覽需依單據類型顯示 PDF 已產生數、Email 待寄送/已寄送/失敗、LINE 未推播/已推播與金額摘要。
- 明細需顯示對象、金額、單號、PDF、Email、LINE、寄送時間、連結與備註。
- 收據總覽不得列入先前錯誤回補、實際 PDF 為繳費單的假收據紀錄。
## 單據版本管理驗收

- 「已產生單據」頁面必須是只讀總覽，不可出現可執行的作廢、補發、重新產生、寄送或確認寫入按鈕。
- 總覽需顯示作廢、補發、重新產生的基本規則，讓行政人員知道目前尚未開放直接執行。
- 作廢規則：作廢後不可再列入 Email 或 LINE push 可寄送清單；原 PDF、金額、寄送狀態與操作紀錄仍需保留。
- 補發規則：補發只能重寄同一份 PDF；不可改金額、內容、收件人或單據編號。
- 重新產生規則：尚未寄送的單據可重產 PDF；已寄送或已 LINE push 的單據需先作廢原單，再建立新版。
- 後續若開放任何作廢、補發、重新產生功能，必須符合「預覽 → 確認 → 執行」。
## LINE push 前置檢查驗收

- 「已產生單據」需顯示 LINE 可推播、缺 LINE ID、未推播、已推播數量。
- 繳費單與收據需以學生/家長 LINE ID 作為推播對象，不可使用講師 LINE ID。
- 領據需以講師名單 B 欄 LINE User ID 作為推播對象，不可使用學生資料。
- 缺 PDF 的單據不可列為可推播。
- 缺 LINE User ID 的單據不可列為可推播，且需顯示警示。
- 作廢單據不可列為可推播。
- 目前總覽頁仍為只讀，不可出現正式 LINE push 確認按鈕。
## LINE push 正式寄送驗收

- 行政端需可分別預覽 `寄送繳費單 LINE`、`寄送收據 LINE`、`寄送領據 LINE`。
- 預覽清單需顯示對象、金額、單據編號、PDF、LINE ID 狀態與警示。
- 只有有 PDF、有 LINE ID、未作廢、LINE 狀態為 `未推播` 的項目可勾選。
- 確認推播後，系統需傳送純文字訊息與 PDF 連結。
- 推播成功後，`單據紀錄表` M 欄需更新為 `已推播`，N 欄需寫入寄送時間，P 欄需追加 LINE push 備註。
- 推播失敗時，M 欄需記錄失敗狀態，P 欄需保留失敗原因。
- 收據 LINE push 不可讀到繳費單 PDF；領據 LINE push 不可讀到學生資料。
- 若 `單據紀錄表` 缺領據紀錄，但 `鐘點結算表` 已有領據 PDF 與單據編號，領據 LINE push 預覽需列出該講師。
- 使用鐘點結算表補讀資料執行 LINE push 時，需先回補 `單據紀錄表`，再更新 LINE 狀態。
- 工具區需看得到 `作廢單據`、`補發單據`、`重新產生單據` 三個入口。
- `作廢單據` 需可預覽有效單據、勾選、二次確認，並要求輸入作廢原因。
- 作廢成功後，`單據紀錄表` O 欄需為 `作廢`，P 欄需追加作廢時間與原因，Q 欄需記錄操作人；PDF 連結與原資料列不可刪除。
- 作廢後該單據不可再列入 Email 或 LINE push 可執行清單。
- `補發單據` 需可預覽有效且有 PDF 的繳費單、收據、領據；作廢單據不可補發。
- 補發需支援 `Email`、`LINE`、`both` 三種方式；補發成功後需追加備註並回寫對應 Email/LINE 狀態與寄送時間。
- 補發不可改金額、內容、PDF 連結或單據編號。
- `重新產生單據` 需可預覽繳費單、收據、領據，且只有未 Email 寄送、未 LINE 推播、未作廢、有 PDF 的項目可勾選。
- 重新產生前需二次確認並要求輸入原因；原因空白時不可執行。
- 重新產生成功後，舊 PDF 需移至垃圾桶，新 PDF 連結需回寫單據來源與 `單據紀錄表`，K 欄產生狀態需為 `重新產生`，P 欄需追加重產時間、原因與舊 PDF 連結，Q 欄需記錄操作人。
- 已 Email 寄送或已 LINE 推播的單據不可重新產生；畫面需提示需先作廢再建立新版。
