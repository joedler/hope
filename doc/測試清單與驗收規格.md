# 測試清單與驗收規格

最後更新：2026-06-05

## 通用驗收原則

- 所有會寫入、產生單據、寄送 Email、LINE push 的功能，都必須先顯示預覽。
- 未確認前不可寫入。
- 權限不足者不可看到或執行行政功能。
- 前端擋住之外，後端也必須檢查權限。
- 錯誤情境必須有明確提示，不可只有無反應。

## LIFF 講師端

### 我的狀態

- 顯示身份別。
- 顯示權限。
- 顯示本月已登記授課。
- 顯示待核銷預排。
- 顯示可登記學生。
- 顯示可登記課程。
- 點選數字後顯示對應資料。

### 授課登記

正常流程：

- 選學生後，只出現該學生課程。
- 日期、時間、課程完整後可送出。
- 成功後寫入授課紀錄。

錯誤流程：

- 空欄位不可送出，需顯示提示。
- 未來時間不可登記為已授課。
- 結束時間早於開始時間不可送出。
- 課程時數過長不可送出。
- 同批新增時間重疊不可送出。
- 同老師同時段重疊不可寫入。
- 同學生同時段重疊不可寫入。

### 預排課程

正常流程：

- 可登記未來課程。
- 成功後寫入預排紀錄。
- 成功後出現在預排核銷清單。

錯誤流程：

- 過去時間不可預排。
- 完全重複預排不可重複寫入。
- 同老師、同學生、同時段重疊需提示。

### 預排核銷

正常流程：

- 未來預排可顯示在清單。
- 課程時間結束後可核銷。
- 有上課後轉入或更新授課紀錄。
- 未上課後取消預排並進入帳務規則。

錯誤流程：

- 未來時間不可核銷。
- 同一筆預排核銷後，再點一次應被擋下。
- 無權限或非本人資料不可核銷。

## 行政端

### 試算

- 學費試算先顯示預覽。
- 鐘點試算先顯示預覽。
- 未確認不可寫入。
- 確認後才寫入試算結果。
- 重複試算不可產生重複錯誤資料。
- 在正式預覽 API 接上前，試算入口不可直接呼叫寫入型後端任務。
- 前端預覽卡需顯示資料狀態與提醒；停用狀態不得取代明細核對。
- 切換行政 Tab 後，上一個預覽卡應清除。
- `adminPreview` API 必須是只讀 API，不可寫入任何試算或單據資料。
- `adminPreview` API 權限不足時需回傳錯誤，不可提供行政預覽。
- `adminPreview` API 可回傳資料來源統計，但不得產生正式結算資料。
- 前端預覽卡需顯示資料來源統計區塊。
- 財務預覽文字需保留換行並分段顯示，至少區分姓名、金額、課程/明細、單號、狀態與提醒。
- 資料來源統計需區分 `ok`、`empty`、`missing`、`error` 四種狀態。
- 當指定月份無資料時，前端需顯示「本月無資料」，不可只顯示 `0`。
- 當 Google Sheets 缺少分頁時，前端需顯示「找不到分頁」，不可被誤認為正常 `0` 筆。
- 當讀取資料失敗時，前端需顯示「讀取失敗」，不可繼續進入寫入或寄送流程。
- 學費試算預覽需顯示學生/課程層級的只讀彙總，不可寫入學費結算表。
- 學費試算預覽需納入預收下月預排、預收核對差異、後收實上課程與帳務補救金額。
- 學費試算預覽若有未核銷預排，需顯示為待核銷並暫不計入退費；待核銷課程列在寫入與產生單據模式下都不可勾選。
- 學費試算確認寫入後端需只接受已勾選且可結算的學生/課程；待核銷預排即使被前端偽造送出也不可寫入。
- 當指定月份已有學費結算資料時，預覽需顯示「回看舊單應使用已產生單據，不要重新試算」類型提醒。
- 當指定月份已有學費結算資料，且授課紀錄中出現同月份尚未標記學費結算的新資料時，預覽需列出該筆資料，狀態為「本月新增未結算，可追加寫入學費結算」，且只允許在寫入學費結算步驟勾選。
- 當指定月份已有學費結算資料時，追加寫入不得再次納入已在該處理月份反映過的帳務補救補收或退費；同一補救金額不可因追加結算重複計入。
- 學費試算確認寫入需經二次確認，後端需重新試算並寫入學費結算表。
- 學費試算確認寫入需阻擋同月份同學生同課程重複寫入。
- 學費試算確認寫入後，授課紀錄與預排紀錄只可回溯標記已寫入的學生/課程，不可整月份全部標記。
- 學費試算確認寫入後，收費單編號必須使用 `R_YYYY_MM_###`，例如 `R_2026_06_001`，不可使用民國年短碼如 `R1506_001`。
- 學費結算表 A 欄應為結算月份，B 欄應為學生姓名，J 欄應為收費單/單據編號。
- 學費結算表 E 欄應為課程日期/時間(金額)；帳務補救補收也必須顯示原上課日期與時間，不可只顯示 `本月帳務補救` 或補收狀態。
- 學費結算表 F 欄才放計算明細，例如實上時數、費率、帳務補救金額。
- 學費試算確認寫入完成後，繳費單預覽需讀到新增的學費結算資料。
- 鐘點試算預覽需顯示講師層級的只讀彙總，不可寫入鐘點結算表。
- 鐘點試算預覽需納入待試算授課、講師時數、應付總額、扣繳稅額、補充保費與實發金額。
- 當指定月份已有鐘點結算資料時，預覽需顯示「回看舊領據或結算應使用已產生單據 / 領據紀錄，不要重新試算」類型提醒。
- 當指定月份已有鐘點結算資料，且授課紀錄中出現同月份尚未標記鐘點結算的新資料時，預覽需列出該筆資料，狀態為「本月新增未結算，可追加寫入鐘點結算」，且只允許在寫入鐘點結算步驟勾選。
- 當指定月份已有鐘點結算資料時，追加寫入不得再次納入已在該處理月份補發過的帳務補救鐘點；同一補發金額不可因追加結算重複計入。
- 鐘點試算確認寫入需經二次確認，後端需重新試算並寫入鐘點結算表。
- 鐘點試算確認寫入需阻擋同月份同講師重複寫入。
- 指定月份已有鐘點結算資料時，鐘點試算預覽需顯示既有結算摘要，確認寫入按鈕不可按；後端直接呼叫確認寫入也必須回錯誤，不可新增列。
- 鐘點試算確認寫入後，領據編號必須使用 `A_YYYY_MM_###`，例如 `A_2026_06_001`，不可使用 `P1506_001`。
- 鐘點結算表 A 欄應為結算月份，B 欄應為講師姓名，C 欄為課程/學生，D 欄為課程日期/時間(金額)，E 欄為時數/費率，F 欄為單科試算，G 欄為應付小計，H 欄為單據編號；補發註記應寫入 J 欄備註。
- 鐘點補發資料需拆成 C 欄課程/學生、D 欄日期時間金額、E 欄時數/費率、F 欄單科試算，不可整段文字擠在同一欄。
- 鐘點試算確認寫入完成後，領據預覽需讀到新增的鐘點結算資料。
- 追加結算後，繳費單/領據編號需沿用當月流水號遞增，不可覆蓋既有單據。
- 重複寫入防呆需以明細層級判斷：學費至少包含學生、課程、課程日期時間明細；鐘點至少包含講師、課程/學生、課程日期時間明細。

### 預排紀錄欄位語意

- 預排紀錄 J 欄為核銷狀態，K 欄為學費結算標記月份，兩者不可混用。
- 同樣是 `未核銷` 的預排，K 欄有月份代表已納入該月預收，K 欄空白代表尚未納入學費結算。
- 學費流程預覽需顯示待核銷預排是否已納入預收，避免行政人員誤以為核銷狀態不同。

### 繳費單

- 產生前顯示預覽。
- 預覽需包含學生、課程、月份、預收/後收、補收/退費、總金額。
- 確認後才產生繳費單。
- 寄送 Email 前需顯示收件人與內容摘要。
- LINE push 前需確認家長 LINE 綁定。
- 已產生單據可查看。
- 後續測試重新產生、作廢、補發。
- 繳費單預覽需讀取學費結算表，顯示學生、單據編號、總金額、課程數與 PDF 產生狀態。
- 繳費單預覽不可重新試算學費，不可產生 PDF，不可寫入 PDF URL 或狀態欄。
- 繳費單預覽不可寄送 Email 或 LINE push。
- 繳費單預覽需由後端回傳 `canConfirm` 與 `confirmAction` 控制確認按鈕，前端不可只用功能名稱開放高風險按鈕。
- 繳費單預覽需回傳完整列；已產生、缺單據編號、缺金額者在產生繳費單模式下不可勾選。
- 繳費單確認產生只可處理勾選學生；未勾選者不可產生。
- 繳費單確認產生需經二次確認，產生 PDF 後回寫學費結算表 PDF URL 與狀態。
- 繳費單確認產生不可寄送 Email 或 LINE push。
- 若該月份繳費單已全部產生，確認按鈕需停用，不可重複產生 PDF。

### 收據

- 產生前顯示預覽。
- 收據預覽需讀取學費結算表的學生、金額、單據編號、收款方式、收據類別與收款日期；收據 PDF 與寄送狀態必須讀取 `單據紀錄表` 中真正的收據紀錄。
- 確認後才產生收據。
- 寄送 Email 前需確認收件人。
- LINE push 前需確認家長 LINE 綁定。
- 一般收據需測入會費、常年會費、捐款。
- 一般收據預覽需讀取一般收據紀錄，顯示姓名、類別、金額、編號與 PDF 狀態。
- 在正式預覽 API 接上前，收據與一般收據入口只可顯示預覽或暫停提示，不可直接產生或寄送。

### 領據

- 產生前顯示預覽。
- 預覽需包含講師、月份、授課紀錄、鐘點費、補充項目。
- 領據預覽需讀取鐘點結算表，顯示講師、應付總額、扣繳稅額、補充保費、實發金額、領據編號、PDF 與寄送狀態。
- 確認後才產生領據。
- LINE push 前需確認講師 LINE 綁定。
- 領據 PDF 產生已接上正式預覽確認流程：可勾選尚未產生 PDF 且有領據編號的講師，確認後產生 PDF 並回寫「待寄送」。
- 領據產生不可自動寄送 Email 或 LINE push；寄送需另走預覽與二次確認流程。
- 學費收據 PDF 產生已接上正式預覽確認流程：可勾選尚未產生 PDF，且已有單據編號、收款方式、收據類別、收款日期與金額的學生，確認後產生 PDF 並回寫「待寄送」。
- 收據產生不可自動寄送 Email 或 LINE push；寄送需另走預覽與二次確認流程。

### 帳務補救

- 可選補收或退費。
- 可選原錯誤月份與處理月份。
- 選學生後，只出現該學生課程。
- 選課程後，自動帶出單價。
- 選時間後，自動計算時數。
- 可選關聯原單號。
- 建立前需顯示補救預覽。
- 確認後才寫入補救紀錄。
- 不可直接改舊繳費單。
- 學費試算預覽與正式寫入需納入處理月份的帳務補救補收/退費。
- 學費試算預覽中的帳務補救需明確標示補收/退費、原月份、上課日期、開始/結束時間、課程名稱、時數、單價、金額、原單號、狀態與原因。
- 補救明細/補收通知預覽只可列出補收、待處理、尚未有補收單號的資料。
- 補救明細/補收通知確認產生後需產生 PDF，單號格式為 `ADJ_YYYY_MM_###`。
- 補救明細/補收通知產生後需回寫補收單號、補收 PDF、補收單狀態，並將原補救紀錄狀態改為已產生補收單。
- 補救明細/補收通知產生不可修改原繳費單、原單號或原 PDF。
- 非行政不可預覽或產生補救明細/補收通知。
- 補救明細/補收通知產生不可自動寫入新的鐘點結算、收據或領據。
- 帳務補救若為補收且有時數、可對應課程設定中的講師，處理月份的鐘點試算預覽需顯示補發鐘點與金額。
- 鐘點試算預覽中的帳務補救補發需明確標示原月份、原上課日期、開始/結束時間、學生、課程名稱、時數與補發金額。
- 鐘點試算確認寫入時，補發鐘點需寫入處理月份的鐘點結算表，不回頭修改原錯誤月份。
- 若處理月份已經有鐘點結算列，後續追加結算只處理新增且尚未標記鐘點結算的授課紀錄；已補發過的帳務補救不得再次自動加入。
- 已有學費結算資料的月份，學費試算預覽需顯示既有結算摘要，不可重新計算授課、預排或帳務補救差異。
- 已有學費結算資料的月份，學費試算預覽文字需簡化並明確顯示「已寫入」；同月份未核銷預排仍需列出，狀態為「待核銷，未寫入」。
- 已有鐘點結算資料的月份，鐘點試算預覽文字需簡化並明確顯示「已寫入」，不得顯示成可重新試算寫入。
- 歷史補救已退款或已結清的月份，回看時不可再顯示舊差異造成的欠款或負數應收。
- GAS manifest 必須包含 `https://www.googleapis.com/auth/documents`，否則產生補收 PDF 時會因 `DocumentApp.openById` 權限不足失敗。

## 時間欄位

- 手機顯示區為 `HH:MM`。
- 電腦顯示區為 `HH:MM`。
- 電腦自訂面板小時為 `06-18`。
- 分鐘為 `00 / 15 / 30 / 45`。
- 不可顯示 `上午`、`下午`、`AM`、`PM` 作為表單顯示值。

## 回歸測試

每次修改 LIFF 後至少測：

- 講師身份登入。
- 行政身份登入。
- 非行政無法進入行政作業。
- 授課登記一筆成功。
- 預排課程一筆成功。
- 未來預排不可核銷。
- 行政高風險功能不可直接寫入。
- 行政端讀取預覽、產生補救明細、建立帳務補救等操作需顯示等待/執行中提示，避免使用者誤以為沒有點選成功。
- 「解除 LINE 帳號綁定」只應出現在我的狀態底部，其他操作頁不可重複顯示。
- 2026-06-11 補充驗收：帳務補救預覽在手機版不可出現空白外框；補救明細文字需保留換行並分段顯示，不得擠成一整行；等待 API 回應時需有可見的讀取提示。
## 帳務工具單據總覽測試

- 行政人員可在帳務工具點選「系統工具」後，再點選「單據總覽」並看到指定月份總覽。
- 總覽需以 `單據紀錄表` 為主要來源，包含繳費單、補收通知、收據、領據、一般收據的筆數、PDF 狀態、Email 狀態、LINE push 狀態與金額摘要。
- 若指定月份 `單據紀錄表` 無資料，需明確顯示目前沒有可查看的單據資料，不可誤導為重新試算。
- 作廢、補發、重新產生按鈕需位於單據卡片上方；按鈕只切換處理模式，不可直接執行。
- 非行政人員呼叫 `adminPreview` 的「已產生單據」功能時，必須回覆權限不足。
- 若某一類單據資料表不存在或讀取失敗，總覽需顯示該類讀取失敗，不可讓整個行政頁崩潰。

## 收據與領據 Email 寄送測試

- 寄送收據 Email 與寄送領據 Email 必須先顯示預覽，不可直接寄送。
- 只有「待寄送」、已有 PDF、且有 Email 的項目可以勾選。
- 缺 PDF、缺 Email、已寄送、或狀態不是待寄送的項目需顯示提醒，但在 Email 寄送模式下不可勾選。
- Email 寄送成功後，原結算表與 `單據紀錄表` 皆需更新為已寄送。
- Email 寄送失敗時，`單據紀錄表` 需保留失敗原因，包含缺 Email、PDF 連結錯誤或 Gmail 例外訊息。
- 繳費單 Email 預覽需可列出 `單據紀錄表` 中 `單據類型=繳費單` 且待寄送的資料。
- 繳費單 Email 可寄送項目需有 PDF、有效 Email、狀態為待寄送。
- 繳費單 Email 寄送成功後，`單據紀錄表` Email狀態需更新為已寄送，並寫入寄送時間與備註。
- 收據 Email 預覽只能列出 `單據紀錄表` 中 `單據類型=收據` 的資料，不可列出繳費單。
- 領據 Email 預覽只能列出 `單據紀錄表` 中 `單據類型=領據` 的資料。
- 繳費單產生後，`單據紀錄表` 的 Email 狀態需為 `待寄送`；既有舊資料若為 `未寄送`，前端總覽需以待寄送語意呈現。
- 確認寄送後需回寫「已寄送」；失敗時需回寫失敗原因。
- 未勾選任何可寄送項目時，不可執行寄送。
- 非行政人員不可呼叫寄送預覽或確認寄送。

## 收據收款資料套用測試

- 收據預覽需顯示收款資料套用區：收款方式、收據類別、收款日期。
- 未填完整三欄不可送出。
- 套用後需寫入學費結算表的收款方式、收據類別、收款日期欄位。
- 已產生收據 PDF 的資料不可被批次套用覆蓋。
- 套用後重新讀取收據預覽，符合條件者應可勾選產生收據。
- 非行政人員不可呼叫收款資料套用 API。

## 行政流程入口測試

- 行政端 TAB 需顯示：學費流程、收據流程、鐘點流程、帳務工具。
- 點選學費流程分頁需直接載入學費試算預覽；流程內不得再顯示 `學費試算` 入口卡。
- 點選收據流程分頁需直接載入收據預覽；流程內不得再顯示 `收款資料` 入口卡。
- 點選鐘點流程分頁需直接載入鐘點試算預覽；流程內不得再顯示 `鐘點試算` 入口卡。
- 行政流程分頁不得再出現 `已產生單據` 按鈕；單據查核與版本管理應走工具區的作廢、補發、重新產生流程，或保留後端只讀總覽供必要時查核。
- 不可再顯示舊的「試算後續流程」或獨立「繳費單」TAB，避免同功能多處重複。
- 流程入口只顯示主要步驟名稱，不得以「可執行」或「只讀」作為入口說明。
- 預覽區需顯示完整明細卡；動作按鈕先切換處理模式，只有符合該模式的項目可勾選。
- 入口只負責導向預覽，不可繞過預覽確認直接寫入、產生或寄送。

## 單據通知與 fallback 測試

- 若 `單據紀錄表` 有繳費單、收據、領據紀錄，寄送通知需以該表為主要來源。
- 寄送通知卡片需同時顯示 Email 與 LINE 子通道，各自顯示收件資料、狀態、可勾選與不可勾選原因。
- 若 `單據紀錄表` 缺繳費單紀錄，但 `學費結算表` 已有繳費單 PDF，繳費單通知預覽需顯示該學生。
- 若 `單據紀錄表` 缺收據紀錄，即使 `學費結算表` 有繳費單 PDF，收據通知預覽也不可顯示該學生；需先完成產生收據 PDF。
- 若 `單據紀錄表` 缺領據紀錄，但 `鐘點結算表` 已有領據 PDF 且狀態為待寄送，領據通知預覽需顯示該講師。
- 使用 fallback 顯示的資料執行通知後，需回補 `單據紀錄表`，並更新 Email/LINE 各自狀態、寄送時間與備註。
- fallback 資料若原結算表狀態為已寄送或已推播，預覽需列入統計，但不可重複勾選對應通道。
- 通知確認端必須使用與預覽端相同的狀態正規化規則，避免預覽可勾選但送出時找不到待寄送資料。

## 已產生單據總覽顯示測試

- 單據總覽位於帳務工具內；可勾選單據後執行作廢、補發、重新產生，但不得顯示在三大流程分頁中。
- 總覽需依單據類型顯示 PDF 已產生數、Email 待寄送/已寄送/失敗、LINE 未推播/已推播與金額摘要。
- 明細需顯示對象、金額、單號、PDF、Email、LINE、寄送時間、連結與備註。
- 收據總覽不得列入先前錯誤回補、實際 PDF 為繳費單的假收據紀錄。
## 單據版本管理驗收

- 「單據總覽」需作為帳務工具內的工作面板，作廢、補發、重新產生按鈕位於單據卡片上方。
- 作廢、補發、重新產生需依 row.actions 啟用或停用；不符合目前操作的單據不可勾選。
- 作廢規則：作廢後不可再列入 Email 或 LINE push 可寄送清單；原 PDF、金額、寄送狀態與操作紀錄仍需保留。
- 補發規則：補發只能重寄同一份 PDF；不可改金額、內容、收件人或單據編號。
- 重新產生規則：尚未寄送的單據可重產 PDF；已寄送或已 LINE push 的單據需先作廢原單，再建立新版。
- 作廢、補發、重新產生必須符合「預覽 → 選動作 → 勾選 → 確認 → 執行」。
## LINE push 前置檢查驗收

- 「單據總覽」需顯示 LINE 可推播、缺 LINE ID、未推播、已推播數量。
- 繳費單與收據需以學生/家長 LINE ID 作為推播對象，不可使用講師 LINE ID。
- 領據需以講師名單 B 欄 LINE User ID 作為推播對象，不可使用學生資料。
- 缺 PDF 的單據不可列為可推播。
- 缺 LINE User ID 的單據不可列為可推播，且需顯示警示。
- 作廢單據不可列為可推播。
- 單據總覽中的寄送狀態只供查核；正式寄送仍走三大流程的「寄送通知」。
## 寄送通知正式寄送驗收

- 行政端需可分別預覽 `寄送繳費單通知`、`寄送收據通知`、`寄送領據通知`。
- 預覽清單需顯示對象、金額、單據編號、PDF、Email 狀態、LINE ID 狀態與警示。
- Email 子通道只有有 PDF、有 Email、Email 狀態為 `待寄送` 的項目可勾選。
- LINE 子通道只有有 PDF、有 LINE ID、未作廢、LINE 狀態為 `未推播` 的項目可勾選。
- 確認寄送通知後，Email 與 LINE 需依卡片子勾選分別寄送；LINE 需傳送純文字訊息與 PDF 連結。
- 推播成功後，`單據紀錄表` M 欄需更新為 `已推播`，N 欄需寫入寄送時間，P 欄需追加 LINE push 備註。
- 推播失敗時，M 欄需記錄失敗狀態，P 欄需保留失敗原因。
- 收據 LINE push 不可讀到繳費單 PDF；領據 LINE push 不可讀到學生資料。
- 若 `單據紀錄表` 缺領據紀錄，但 `鐘點結算表` 已有領據 PDF 與單據編號，領據 LINE push 預覽需列出該講師。
- 使用鐘點結算表補讀資料執行 LINE push 時，需先回補 `單據紀錄表`，再更新 LINE 狀態。
- 工具區需看得到 `作廢單據`、`補發單據`、`重新產生單據` 三個入口。
- `作廢單據` 需可預覽有效單據、勾選、二次確認，並要求輸入作廢原因。
- 作廢成功後，`單據紀錄表` O 欄需為 `作廢`，P 欄需追加作廢時間與原因，Q 欄需記錄操作人；PDF 連結與原資料列不可刪除。
- 作廢後該單據不可再列入 Email 或 LINE push 可執行清單。
- `補發單據` 需可預覽有效且有 PDF 的繳費單、收據、領據；作廢單據不可補發。
- 補發需支援 `Email`、`LINE`、`both` 三種方式；補發成功後需追加備註並回寫對應 Email/LINE 狀態與寄送時間。
- 補發不可改金額、內容、PDF 連結或單據編號。
- `重新產生單據` 需可預覽繳費單、收據、領據，且只有未 Email 寄送、未 LINE 推播、未作廢、有 PDF 的項目可勾選。
- 重新產生前需二次確認並要求輸入原因；原因空白時不可執行。
- 重新產生成功後，舊 PDF 需移至垃圾桶，新 PDF 連結需回寫單據來源與 `單據紀錄表`，K 欄產生狀態需為 `重新產生`，P 欄需追加重產時間、原因與舊 PDF 連結，Q 欄需記錄操作人。
- 已 Email 寄送或已 LINE 推播的單據不可重新產生；畫面需提示需先作廢再建立新版。

## 單據 PDF 檔名與前端顯示測試

- 新產生的繳費單 PDF 檔名需以收費單號開頭，例如 `R_2026_06_001_繳費單_學生姓名.pdf`。
- 新產生的收據 PDF 檔名需以單據編號開頭，例如 `R_2026_06_001_收據_學生姓名.pdf`。
- 新產生的領據 PDF 檔名需以領據編號開頭，例如 `A_2026_06_001_領據_講師姓名.pdf`。
- 新產生的一般收據 PDF 檔名需以一般收據編號開頭。
- 若資料夾已有舊檔名 PDF，查詢、產生防呆與重新產生流程需能相容舊檔名，不可造成既有單據失聯。
- Drive 資料夾維持現有分類，不要求每月自動新增子資料夾。
- 行政預覽明細卡片不可顯示完整 Drive 長網址；需顯示短狀態，例如 `PDF：已產生` 或 `連結：已產生`，避免手機版破格。
- 若後端回傳其他長網址，前端仍需斷行或替換為短狀態，不可超出卡片寬度。

## Script Properties 測試

- 缺少 `LINE_CHANNEL_TOKEN`、`SPREADSHEET_ID`、`LEAVE_SHEET_ID`、`REPORT_SHEET_ID`、`SHEET_ID_MEMBER` 任一必填屬性時，系統應明確報錯。
- 未設定 `LINE_GROUP_ID`、`ADMIN_LINE_USER_IDS`、單據範本 ID 或 PDF 資料夾 ID 時，既有正式功能仍應使用 fallback 正常運作。
- 設定 `ADMIN_LINE_USER_IDS` 後，行政權限應改讀 Script Properties；換行、逗號、分號分隔都需可解析。
- 設定單據範本與 PDF 資料夾屬性後，新產生 PDF 應使用屬性指定的範本與資料夾。
- `forceAuth()` 必須能開啟一般收據範本並觸發文件權限授權。

## 行政端手機 UI 驗收

- 行政流程按鈕在手機寬度下不可文字溢出或互相重疊。
- 預覽區不可出現三層以上明顯外框包覆。
- PDF 連結、備註、明細長文字需換行或轉成短狀態，不可超出卡片。
- `submitting` 為真但沒有作業訊息時，不可顯示空白等待框。
- 行政頁視覺應維持淺色、公益團體友善風格，不回到深色管理後台風格。

## 帳務補救防重與 PDF 連結驗收

- 使用 LIFF 帳務補救建立一筆資料後，再以相同學生、課程、原上課日期、開始時間、結束時間、補救類型送出，系統需阻擋並顯示明確錯誤。
- 使用 LINE 文字指令補登相同資料時，也需阻擋重複建立。
- 已作廢或已取消的舊帳務補救紀錄不應阻擋重新建立。
- 補登成功後，頁面下方需顯示最近補登結果，並可點選查看本月補救明細/補收通知。
- `補救明細/補收通知` 預覽需位於帳務補救表單後方，避免操作者誤以為送出無反應。
- 繳費單、收據、領據、Email、LINE、作廢、補發、重新產生、已產生單據等行政明細列，若後端有 PDF URL，前端需顯示 `查看 PDF` 文字連結。
- 前端不可把完整 Drive URL 直接顯示在卡片內容中造成破格。

## 一般收據 LIFF 驗收

- 行政人員可在 `帳務工具` 點選 `一般收據` 並看到填寫表單。
- 表單需包含收據日期、姓名/單位、身分證號/統一編號、Email、類別、收款方式與金額。
- 未填日期、姓名/單位或有效金額時，不可送出。
- 送出前需出現二次確認，確認內容需包含日期、姓名/單位、身分證號/統一編號、Email、類別、收款方式與金額。
- 確認後需產生一般收據 PDF，並寫入 `一般收據紀錄`。
- `一般收據紀錄` K 欄需寫入身分證號/統一編號；L 欄需寫入 Email收件人。
- Email 空白或格式錯誤時，前端不得送出，後端 API 也必須拒絕寫入。
- 確認後需同步寫入 `會計日記帳` 借貸分錄。
- 產生後需刷新一般收據預覽，並在明細列顯示 `查看 PDF`。
- LIFF 一般收據產生後不可自動寄送 Email 或 LINE push。
- 一般收據 Email 需在一般收據同頁清單中可預覽、勾選、二次確認後寄送，不需另開第二個寄送入口。
- 一般收據 Email 寄送成功後，`一般收據紀錄` I 欄需更新為 `已寄送`。
- 一般收據 Email 缺 PDF 或缺 Email 時需顯示警示，但在 Email 寄送模式下不可勾選。
- 一般收據 Email 寄送優先讀取 `一般收據紀錄` L 欄，缺值時才回查會員/捐款者資料。
- 捐款稅務同步需優先讀取 `一般收據紀錄` K 欄，缺值時才回查會員/捐款者資料；仍缺身分證號/統一編號時需列入排除名單。
- 捐款稅務同步只處理類別為 `捐款` 的一般收據；入會費與常年會費不得進入捐款申報檔。
- 一般收據 LINE push 不開放，前端不可出現可執行入口。
- 既有 LINE `開一般收據` 流程仍需可正常使用，且與 LIFF 共用同一套寫入邏輯。

## 行政端入口與預覽簡化驗收

- 行政端 TAB 只顯示：學費流程、收據流程、鐘點流程、帳務工具。
- 不得再顯示獨立的「帳務補救」TAB 或「工具」TAB。
- 「帳務工具」首頁顯示四項：稅務專區、帳務補救、一般收據、系統工具。
- 帳務補救表單不可在帳務工具首頁直接展開，需點選「帳務補救」後才顯示。
- 單據總覽有單據資料時不得顯示大段摘要或規則說明，只顯示單據狀態卡片。
- 作廢、補發、重產不得作為帳務工具上方入口；需顯示在單據總覽資料卡片上方，依勾選單據狀態啟用或停用。
- 系統工具需包含單據總覽與查我的 ID。
- 稅務專區點選後只顯示 `建置中`，不得出現暫緩處理或測試期說明文字。
- 行政人員不得看到「目前沒有可登記的學生」或「目前沒有可登記的課程」等講師端課程設定提示。
- 月份選擇器需出現在目前流程工作區內，不再獨立放在行政頁最上方。
- 單據總覽只放在「帳務工具」內；學費、收據、鐘點流程不得重複顯示總覽入口。
- 行政預覽有學生或講師明細列時，畫面不得顯示上方大段摘要、資料來源統計卡、目前狀態提示或禁用確認按鈕。
- 行政預覽不得出現開發期文字：`預覽模式`、`後端只讀預覽`、`尚未開放`、`本機預覽`、`目前狀態`。
- 明細列姓名前需顯示 `●`，課程、日期時間、金額、PDF 與狀態需分行呈現，手機版不得被長文字或 Drive URL 撐破。

## 三大行政流程統一操作驗收

- 學費、收據、鐘點分頁被點選時，需直接載入該分頁主要預覽，不得再要求行政人員多點一次主入口卡。
- 學費流程不得顯示 `學費試算`、`產生繳費單`、`寄送繳費單`、`LINE push` 等入口卡片。
- 收據流程不得顯示 `收款資料`、`產生收據`、`寄送收據`、`LINE push`、`一般收據` 等入口卡片；一般收據需移至帳務工具。
- 鐘點流程不得顯示 `鐘點試算`、`產生領據`、`寄送領據`、`LINE push` 等入口卡片。
- 三大流程不得出現 `可執行`、`預覽後確認`、`Email 預覽確認`、`傳 PDF 連結` 等卡片說明文字。
- 資料卡片上方需顯示該流程可用動作按鈕；不得把主要動作藏在卡片列表下方。
- 學費流程動作按鈕需包含：寫入學費結算、產生繳費單、寄送通知。
- 指定月份已有學費結算時，學費試算預覽需顯示繳費單狀態與 `查看 PDF` 連結，不得只顯示已寫入摘要。
- 待核銷預排需顯示於學費試算預覽，狀態為 `待核銷，未寫入`；在寫入、產生、寄送模式下 checkbox 需停用。
- 所有預覽資料列都需以一致卡片顯示黑點、狀態與提醒；checkbox 是否可用需依目前選定動作即時判斷。
- 不符合目前動作的卡片不得顯示可點擊感強烈的 checkbox，需顯示不可處理標示與短原因。
- Email / LINE 狀態不得在主狀態、明細與子通道中重複顯示；主狀態保留流程/PDF 狀態，Email / LINE 只在子通道區顯示。
- 流程動作按鈕需先切換處理模式，不可直接執行；確認按鈕只在已勾選該模式可處理資料時啟用。
- 三大流程進入分頁後需直接顯示完整狀態卡；寄送通知不得另開查詢頁，寫入、產生、寄送按鈕都必須依目前卡片 row.actions 啟用或停用。
- 收據流程動作按鈕需包含：套用收款資料、產生收據、寄送通知。
- 收據已具備收款方式、收據類別、收款日期時，不得再於「套用收款資料」模式下勾選；若尚未產生收據 PDF，才可於「產生收據」模式勾選。
- 收據已產生 PDF 時，不得再於「套用收款資料」或「產生收據」模式下勾選，只能依 Email/LINE 狀態進入寄送通知。
- 鐘點流程動作按鈕需包含：寫入鐘點結算、產生領據、寄送通知。
- 鐘點已寫入結算後，鐘點試算預覽仍需顯示講師 rows；已產生領據 PDF 者不得再於「產生領據」模式下勾選。
- `checkbox` 與動作按鈕數量需依 row.actions 的對應動作判斷，不得用單一 `selectable` 套用所有流程。
- 寄送通知卡片需顯示 Email 與 LINE 子勾選；缺 Email 時只停用 Email，缺 LINE ID 時只停用 LINE，不可互相阻擋。
- 確認寄送通知時，Email 與 LINE 需各自回寫狀態，並回傳成功、失敗與略過明細。
- 單據總覽不得顯示大段操作規則文字；有單據資料時以單據卡片為主，作廢、補發、重新產生按鈕需位於卡片上方並依 row.actions 判斷可用狀態。
