# 使用者視角需求與測試環境改版清單

更新日期：2026-06-02

## 1. 目前共識

空空BOT 已自 2026 年 1 月開始正式使用，1 至 5 月的授課、請假、財務與單據相關資料已累積在正式 Google Sheets 中。因此本次改版不應推翻重做正式系統，而應採用：

- 保留正式 Google Sheets 既有資料與欄位邏輯。
- 保留既有 LINE 文字指令作為過渡期 fallback。
- 在現有 repo 基礎上繼續重構。
- 另開測試環境平行開發與驗證。
- 等低峰期確認穩定後，再切換正式 LINE Webhook / LIFF / Rich Menu。

簡單結論：

> 不是重新開一個完全新專案，而是在現有專案基礎上升級；但測試環境必須獨立。

## 2. 目前使用者角色與功能

### 講師

目前功能：

- 授課登錄。
- 預排課程。
- 預排核銷。
- 刪除最後一筆授課或預排紀錄。
- 請假。
- 查詢選單、說明或自身操作資訊。

目前操作方式：

- 主要透過 LINE 文字指令，例如 `登錄`、`預排`、`核銷`、`請假`。
- 系統回覆 Flex Message 選單或卡片。
- 使用者再輸入日期、時間、學生或課程資訊。
- GAS 檢查重疊、計算時數或寫入 Google Sheets。

主要痛點：

- 每月可能只操作一次，容易忘記指令。
- 日期與時間格式容易輸入錯。
- 對話流程中斷後，使用者不知道如何回到正確步驟。
- 使用者難以判斷錯誤來自輸入、資料表還是程式計算。

### 助理 / 行政

目前功能：

- 代替講師登錄、預排、核銷。
- 學費試算。
- 鐘點費試算。
- 產生繳費單。
- 查詢繳費單。
- 開收據、寄收據。
- 開領據、寄領據。
- 發公告。
- 查 ID、查群 ID。

目前操作方式：

- 透過 LINE 文字指令搭配 Flex Message。
- 部分流程使用快取狀態，要求使用者接著輸入月份、姓名或確認操作。

主要痛點：

- 指令多且相似，助理不易記憶。
- 月份、姓名、格式輸入錯會造成流程失敗。
- 財務與單據功能風險較高，不適合直接一鍵寫入或寄送。
- 批次功能若缺少預覽與異常清單，容易在錯誤資料上繼續執行。

### 財務 / 管理者

目前功能：

- 會計記帳。
- 財務三表。
- 免稅試算。
- 年度扣繳。
- 捐款申報。
- 一般收據。
- 批次產生或寄送單據。

主要痛點：

- 計算結果仰賴 Google Sheets 欄位、公式與 GAS 程式一致。
- 高風險操作需要清楚的預覽、確認與操作紀錄。
- 不應讓未授權者看到或執行敏感財務功能。

## 3. 未來建議操作方式

最佳方向為：

> LINE Rich Menu 作為入口，LIFF Web App 作為主要操作介面，LINE 訊息作為通知與確認結果。

### 講師端建議

講師不再需要記文字指令，改由 Rich Menu 進入 LIFF：

- 授課登記。
- 預排課程。
- 預排核銷。
- 請假申請。
- 我的紀錄。
- 說明。

LIFF 表單應提供：

- 學生下拉選單。
- 課程下拉選單。
- 日期選擇器。
- 開始與結束時間選擇器。
- 正式授課 / 預排切換。
- 送出前摘要。
- 後端重疊檢查。
- 成功或失敗訊息。

### 助理 / 行政端建議

行政使用 LIFF 工作台，不再依賴大量文字指令。

建議分區：

- 課程代操作。
- 月結試算。
- 單據作業。
- 講師與學生查詢。
- 公告工具。
- 系統工具。

行政操作流程建議：

1. 選擇月份、講師、學生或任務。
2. 系統顯示試算或操作預覽。
3. 顯示異常資料，例如缺 email、缺金額、缺講師資料。
4. 助理二次確認。
5. 系統才寫入 Sheet、產生 PDF 或寄送 Email。
6. LINE 或 LIFF 顯示執行結果與錯誤明細。

### 財務與高風險功能建議

高風險功能不可第一版就直接一鍵實寫。

建議採用：

- 先試算。
- 再預覽。
- 顯示異常清單。
- 二次確認。
- 寫入或寄送。
- 留操作紀錄。

適用功能：

- 學費試算。
- 鐘點費試算。
- 產生繳費單。
- 寄收據。
- 寄領據。
- 年度扣繳。
- 捐款申報。
- 免稅試算。

### 測試 GAS 專案應如何建立

建議建立「獨立 Apps Script 專案」，不要建在複製 Sheet 的「擴充功能 -> Apps Script」底下。

原因：

- 獨立 GAS 更適合使用 `clasp` 管理與部署。
- 測試 GAS、正式 GAS、測試 Sheet、正式 Sheet 的邊界更清楚。
- 未來切換 Webhook 或部署版本比較乾淨。
- 測試專案可重建、重推、重部署，不會綁死在單一試算表容器內。
- 若之後產品化或複製給其他客戶，獨立 GAS 架構更容易複製。

複製 Sheet 仍然作為測試資料庫，只要把測試 Sheet ID 設定到測試 GAS 的 Script Properties 即可。

## 11. 2026-06-03 完整測試清單

### A. 身分與權限

- 未綁定 LINE 使用者開啟 LIFF，應顯示講師身分認證。
- 正確姓名可綁定講師。
- 已被其他 LINE 帳號綁定的姓名不可重複綁定。
- 一般講師不顯示行政專區。
- Admin 顯示行政專區。
- 解除綁定後，再次開啟需重新認證。

### B. 講師下拉選單

- 有課程的講師開啟 LIFF，學生下拉只顯示該講師學生。
- 選擇學生後，課程下拉只顯示該學生對應課程。
- 切換學生後，已選課程應清空，避免送出舊課程。
- 沒有學生或課程的講師，學生/課程下拉為空並顯示提示。
- 課程設定表 A 欄使用 LINE User ID 或講師姓名時，都能正確抓到資料。

### C. 授課登記

- 正常登記可寫入 `授課紀錄`。
- 空欄位不可送出。
- 結束時間早於或等於開始時間不可送出，且前端顯示提示。
- 找不到課程設定或鐘點單價時不可寫入。
- 同講師同時段重疊不可寫入，且顯示錯誤。
- 同學生同時段重疊不可寫入，且顯示錯誤。
- 同講師、同學生、同課程、同時段完全重複不可寫入。
- `MAGIC_KEYWORDS` 例外課程應允許重疊。

### D. 預排課程

- 正常預排可寫入 `預排紀錄`，狀態為 `未核銷`。
- 同講師同時段預排重疊不可寫入。
- 同學生同時段預排重疊不可寫入。
- 同講師、同學生、同課程、同時段完全重複預排不可寫入。
- 已取消的預排不應阻擋新預排。
- `MAGIC_KEYWORDS` 例外課程應允許重疊。

### E. 預排核銷

- 只顯示狀態為 `未核銷` 的預排。
- 未來預排課程應出現在待核銷清單，讓講師確認預排成功。
- 課程尚未到下課時間時，核銷按鈕應停用，且即使直接打 API 也不可核銷。
- 已下課的預排可按「有上課」核銷，並寫入或更新授課紀錄。
- 已下課的預排可按「未上課」取消。
- 同一筆預排不可重複核銷。
- 非本人且非 Admin 不可核銷他人預排。

### F. 請假

- 正常請假可寫入請假表。
- 請假後剩餘額度應更新。
- 空欄位不可送出。
- 結束時間早於或等於開始時間不可送出，且前端顯示提示。
- 同一講師同日期同時段重複請假不可寫入，且顯示提示。
- 超過可用額度時不可寫入。

### G. 行政學費試算

- LIFF 行政學費試算目前應顯示暫停使用，不可直接寫入。
- LINE 文字指令學費試算仍須先顯示預覽與確認卡。
- 確認寫入後，`學費結算表` A 欄應保留結算月份，B 欄應為學生姓名，J 欄才是單據編號。
- 多課程學生只應在總額列產生單據編號。
- 同月份同學生已有結算紀錄時，應阻擋重複寫入。
- 寫入後應回溯標記 `授課紀錄` 學費結算欄。
- 預收課程應正確處理上月核對、下月預收、取消退費與差額。

### H. 行政鐘點試算

- LIFF 行政鐘點試算目前應顯示暫停使用，不可直接寫入。
- LINE 文字指令鐘點試算仍須先顯示預覽與確認卡。
- 確認寫入後，`鐘點結算表` A 欄應保留結算月份，B 欄應為講師姓名，H 欄才是領據編號。
- 同月份同講師已有結算紀錄時，應阻擋重複寫入。
- 寫入後應回溯標記 `授課紀錄` 鐘點結算欄。
- 扣繳稅額、補充保費、實發金額需依講師設定正確計算。

### I. 單據與寄送

- 產生繳費單前應確認 `學費結算表` 已有正確學生姓名、金額、單據編號。
- 已產生過繳費單的學生不可重複產生，除非行政手動清理舊欄位。
- 寄收據前應確認 PDF 已存在。
- 開領據前應確認 `鐘點結算表` 已有正確講師姓名、金額、領據編號。
- 已產生過領據的講師不可重複產生，除非行政手動清理舊欄位。
- 批次寄信功能正式開放前需先做小範圍測試。

- 檢查 1 至 5 月正式資料是否仍可被查詢與產製單據。
- 檢查測試期間錯寫的 `學費結算表` 欄位，尤其 B 欄誤寫單據編號的列。
- 檢查同月份重複結算列，確認是否需要保留、刪除或標註作廢。
- 建立資料一致性檢查：授課紀錄、預排紀錄、學費結算表、鐘點結算表互相對應。

## 12. 帳務補救機制

### 核心原則

- 已發出的舊繳費通知單不直接覆蓋。
- 錯誤用「調整紀錄」修正，保留原單號、原因、操作人與處理狀態。
- 回看舊月份時，優先查詢已結算快照，不重新試算。
- 真正需要重算時，應走作廢/更正版流程，不直接刪除舊資料。

### 補救類型

- `補收`：講師漏登已上課程，造成原繳費通知短收。
- `退費`：誤登、多登或已收但未上課，造成原繳費通知超收。
- `作廢重開`：已發單但尚未付款或行政確認需整張重開，保留原單並建立新版。

### 第一版指令

第一版先建立 `學費調整紀錄表` 與人工留痕，不會直接更改舊單，也不會自動併入下一期。

```text
帳務補救 說明
帳務補救 補收 YYYY/MM 學生 課程 YYYY/MM/DD 開始 結束 時數 單價 原單號 原因
帳務補救 退費 YYYY/MM 學生 課程 YYYY/MM/DD 開始 結束 時數 單價 原單號 原因
帳務補救 查詢 YYYY/MM 學生
```

範例：

```text
帳務補救 補收 2026/05 均逸 陪伴對話/自學紀錄同行 2026/04/18 1400 1530 1.5 1000 R_2026_04_019 講師漏登
帳務補救 查詢 2026/05 均逸
```

### 後續要接的流程

- 在學費查詢時顯示該學生該月份的調整紀錄。
- 在下一期學費試算時可選擇將 `待處理` 的補收/退費併入。
- 產生補開或更正版繳費通知單。
- 將調整紀錄狀態更新為 `已併入下期`、`已補開單` 或 `已作廢`。
