## 2026-06-04 時間欄位規格

決策：

- 表單時間顯示區一律為 24 小時制 `HH:MM`。
- 分鐘維持 `00 / 15 / 30 / 45`。
- 手機使用原生時間選擇器。
- 電腦使用自訂時間面板。

原因：

- 原生 `input type="time"` 在不同瀏覽器與 LINE WebView 顯示不一致。
- 電腦端可能出現上午/下午與長清單，不適合本系統。
- 使用者重點是欄位顯示區必須清楚、不可誤解。

詳細紀錄：

- 見 `doc/時間欄位設計決策紀錄.md`。

## 2026-06-05 行政端高風險作業原則

決策：

- 所有會寫入、產生單據、寄送 Email、LINE push 的功能，一律採用「預覽 → 確認 → 執行」。
- 行政作業改為 Tab 工作台。
- 帳務補救正式收入行政 Tab。

原因：

- 行政功能會越來越多，以大格子入口呈現會造成頁面過長與操作混亂。
- 月底結算與單據寄送屬高風險作業，必須降低誤寫、誤寄、重複產生。
- 帳務補救涉及舊月份錯誤與家長溝通，必須獨立、可追溯。

## 2026-06-05 單據與寄送功能規格

決策：

- 繳費單：需預覽、產生、寄送 Email、後續 LINE push 給家長、查看已產生單據、後續重新產生/作廢/補發。
- 收據：需預覽、產生、寄送 Email、後續 LINE push 給家長。一般收據處理入會費、常年會費、捐款，至少支援預覽與寄送。
- 領據：需預覽、產生、寄送 Email、後續 LINE push 給講師。

原因：

- Email 與 LINE push 都可能造成對外通知錯誤，因此都必須先預覽。
- 單據生命週期需要狀態管理，不能只靠一次性產生。

## 2026-06-05 行政端 Tab 工作台初版

決策：

- 行政端 UI 從大格子入口改為 Tab 工作台。
- Tab 包含：試算、繳費單、收據、領據、帳務補救、工具。
- 帳務補救正式收入 Tab，不再放在行政頁底部。
- 寄送繳費單、LINE push 繳費單、LINE push 收據、LINE push 領據等尚未接後端 API 的項目，先以提示狀態呈現，不執行實際寫入或寄送。
- 試算、繳費單、收據、領據、一般收據、稅務專區的行政入口先改為前端預覽卡片，不直接呼叫高風險後端任務。

原因：

- 行政功能數量多，平鋪大格容易造成掃描困難與誤操作。
- 高風險功能必須先建立「預覽 → 確認 → 執行」的工作區結構，再逐步接正式 API。
- 在正式預覽 API、收件人確認與二次確認完成前，不應讓 UI 入口直接寫入、產生或寄送。

## 2026-06-05 行政預覽 API 骨架

決策：

- 新增 `adminPreview` LIFF API 作為只讀預覽入口。
- `adminPreview` 回傳標題、月份、摘要、檢查項目、資料來源統計、狀態與下一步文字。
- `adminPreview` 不可寫入試算表、產生 PDF、寄送 Email 或 LINE push。
- 前端預覽優先呼叫 `adminPreview`，失敗時退回本機預覽。
- 資料來源統計不只回傳筆數，也需回傳狀態：有資料、本月無資料、找不到分頁、讀取失敗。

原因：

- 行政端後續需要正式預覽資料，但不能讓預覽流程誤用執行型 `adminTask`。
- 將預覽 API 與執行 API 分離，可降低月底結算誤寫與誤寄風險。
- 單純顯示 `0` 筆無法分辨是本月沒有資料，還是 Google Sheets 分頁缺失或讀取異常。

## 2026-06-05 學費試算只讀預覽

決策：

- 學費試算入口先提供只讀彙總，不開放寫入。
- 只讀預覽需彙總待試算授課、預收下月預排、預收核對差異、後收實上課程與帳務補救金額。
- 若指定月份已有學費結算資料，預覽需提醒行政人員：這可能是回看舊單情境，應改用「已產生單據」，不可重新試算。

原因：

- 先讓行政人員看見接近正式結算的資料，才能開始驗證預收/後收與補救規則。
- 舊月份回看與重新試算必須分流，避免再次產生「回查舊資料卻被重新計算」的錯誤。

## 2026-06-05 鐘點試算只讀預覽

決策：

- 鐘點試算入口先提供只讀彙總，不開放寫入。
- 只讀預覽需彙總待試算授課、講師時數、應付總額、扣繳稅額、補充保費與實發金額。
- 若指定月份已有鐘點結算資料，預覽需提醒行政人員：這可能是回看舊領據或結算情境，應改用「已產生單據 / 領據紀錄」，不可重新試算。
- 領據預覽仍屬下一階段，需等鐘點試算資料確認穩定後再接產生、寄送與 LINE push。

原因：

- 鐘點費與領據對講師付款，屬高風險財務流程，必須先確認試算來源。
- 鐘點試算與領據生命週期要分開，避免尚未確認試算時就進入對外單據流程。

## 2026-06-05 繳費單只讀預覽

決策：

- 繳費單入口先提供只讀預覽，不產生 PDF、不寫入、不寄送。
- 繳費單預覽讀取既有學費結算表資料，顯示學生、單據編號、總金額、課程數與 PDF 產生狀態。
- 繳費單預覽不重新試算學費。若沒有學費結算資料，需先回到學費試算流程。

原因：

- 學費試算與繳費單產生是不同階段，繳費單不能再次改變金額。
- 先確認結算表中的單據資料，才能安全進入產生、補發、寄送與 LINE push。

## 2026-06-10 補救明細/補收通知

決策：

- 帳務補救中的補收/退費不修改原繳費單，金額併入處理月份學費試算與月結。
- 需要補充通知時，可產生補救明細/補收通知，單號格式為 `ADJ_YYYY_MM_###`。
- 補救明細/補收通知只處理調整類型為補收、狀態為待處理、尚未有補收單號的資料。
- 確認產生後回寫學費調整紀錄表：狀態、補收單號、補收 PDF、補收單狀態。
- 原繳費單號只作為關聯，不可覆蓋原 PDF 或原單據資料。

原因：

- 講師漏登或家長回覆錯誤時，行政需要補救機制，但不能破壞已寄出的歷史單據。
- 補救明細可讓家長清楚知道這是追加補收，也保留完整追溯關係；正式收款與收據仍應與處理月份月結對齊。

## 2026-06-10 已結算月份回看不重算

決策：

- 學費試算指定月份若已有學費結算表資料，預覽改為顯示既有結算摘要。
- 已結算月份不再重新抓授課紀錄、預排紀錄或帳務補救紀錄計算差異。
- 補收、退費、漏登等帳務補救維持獨立紀錄，帳務影響歸入處理月份月結，不覆蓋原單據。
- 補救明細/補收通知不自動寫入鐘點結算、收據、領據；相關流程需各自預覽與確認。
- 若帳務補救是漏登已上課且可對應講師與課程，補發鐘點應出現在處理月份的鐘點試算預覽中，經確認後才寫入鐘點結算。

原因：

- 回看舊月份是查詢，不是重新結算。若重新試算，已退款或已結清的歷史差異會再次出現，造成錯誤欠款或負數應收。
- 家長繳費單、講師鐘點、收據、領據、稅務認列的生命週期不同，不能因為新增一張補收單就隱性寫入整串資料；但同一筆漏登補救應在相關預覽中被看見，讓行政逐一確認。

## 2026-06-10 結算單據編號與鐘點欄位

決策：

- 學費收費單/單據編號統一使用 `R_YYYY_MM_###`，例如 `R_2026_06_001`。
- 講師鐘點領據編號統一使用 `A_YYYY_MM_###`，例如 `A_2026_06_001`。
- 編號年月必須使用結算月份，不可使用執行當下日期，也不可使用民國年短碼。
- 鐘點結算表寫入欄位固定依正式表頭：A 結算月份、B 講師姓名、C 課程/學生、D 課程日期/時間(金額)、E 時數/費率、F 單科試算、G 應付小計、H 單據編號、I 存檔時間、J 備註、K 檔案連結、L Email狀態、M 扣繳稅額、N 補充保費、O 實發金額。
- 講師所得格式代號與國籍只作為稅務計算依據，不寫入鐘點結算表 C/D 欄，避免與既有表頭錯位；帳務補救補發等註記寫入 J 欄備註。

原因：

- 原先出現 `R1506_001`、`P1506_001` 等舊格式，會與現行單據規則不一致。
- 鐘點確認寫入曾將 `9B`、`本國人` 寫入結算表前段欄位，造成行政核對、領據預覽與後續 PDF 讀取錯位。
## 2026-06-11 已產生單據先做只讀總覽

決策：
- 行政端先開放「已產生單據」只讀總覽。
- 總覽彙整繳費單、收據、領據、一般收據狀態，供回看與核對。

## 2026-06-11 收據收款資料與試算流程收斂

決策：
- 收據產生前，行政端需可在 LIFF 補填收款方式、收據類別、收款日期。
- 批次套用只處理尚未產生收據 PDF 的學費結算資料，避免覆蓋已產生單據。
- 試算分頁開始加入後續流程入口，讓主要操作逐步集中在試算項下方。

原因：
- 原先收據預覽會顯示「方式未填 / 類別未填 / 日期未填」，但前端沒有可填寫入口，造成流程卡住。
- 月結工作是連續流程，行政人員應從試算結果往後完成單據與寄送，而不是在多個分頁中猜下一步。

## 2026-06-11 單據狀態改由單據紀錄表管理

決策：
- 新增 `單據紀錄表`，集中管理單據 PDF、Email、LINE push、補發、作廢狀態。
- 繳費單、收據、領據產生後，都同步寫入 `單據紀錄表`。
- 三條正式流程的最後一步皆為 LINE push + Email。

原因：
- 舊流程曾直接產生 PDF 到雲端資料夾，未必同步寫入試算表欄位。
- 繳費單與收據共用學費結算表 PDF/狀態欄會造成狀態混淆。
- 先建立單據紀錄表，後續才能安全整併 UI、補發、作廢與 LINE push。

## 2026-06-11 已產生單據改讀單據紀錄表

決策：
- 「已產生單據」總覽改以 `單據紀錄表` 為主，不再分別從學費結算表、鐘點結算表、一般收據紀錄拼湊狀態。
- 收據與領據 Email 寄送結果同步回寫 `單據紀錄表` 的 Email 狀態、寄送時間與備註。
- 原結算表仍保留相容欄位，但補發、作廢、重新產生、LINE push 等後續狀態管理以 `單據紀錄表` 為準。

原因：
- 單據 PDF、Email、LINE push、作廢與補發若分散在多張表，容易造成畫面與實際狀態不一致。
- 後續要做補發、作廢與重新產生時，需要穩定的中心紀錄來源。

## 2026-06-11 Email 預覽依單據類型分流

- 收據 Email 預覽只讀 `單據類型=收據`，領據 Email 預覽只讀 `單據類型=領據`。
- 繳費單 Email 先開放只讀預覽，不直接寄送。

原因：
- 舊流程曾讓繳費單與收據共用學費結算表 PDF/狀態欄，導致收據 Email 預覽誤判繳費單狀態。
- 手機上原生月份欄不一定像選單，需提供明確快速入口。

## 2026-06-11 行政端入口收斂為三條主流程

決策：
- 行政端 TAB 改為學費流程、收據流程、鐘點流程、帳務補救、工具。
- 學費流程集中學費試算、繳費單、寄送繳費單、已產生單據。
- 收據流程集中收款資料、產生收據、寄送收據、一般收據、已產生單據。
- 鐘點流程集中鐘點試算、產生領據、寄送領據、已產生單據。
- 移除舊的「試算後續流程」與獨立繳費單 TAB。

原因：
- 同一功能散在試算區、繳費單 TAB、收據 TAB 與領據 TAB，容易讓行政人員不知道下一步在哪。
- 依照真實工作順序分流，可以降低月底操作錯誤與重複產出風險。

## 2026-06-12 行政端流程狀態提示

決策：
- 流程卡片需標示「可執行」或「只讀」。
- 預覽區下方狀態文字改為依照 `canConfirm`、是否有可勾選項目、是否有確認 API 動態顯示。
- 尚未開放的 LINE push 不以可執行入口呈現。

原因：
- 行政人員需要在按下功能前就知道下一步是核對、寫入、產生 PDF、寄送或尚未開放。
- 泛用的「尚未開放」提示會讓已接上的功能也看起來不能用，增加測試混亂。

## 2026-06-12 繳費單 Email 開放正式寄送

決策：
- 繳費單 Email 改為可勾選寄送，流程與收據/領據 Email 一致：預覽、勾選、二次確認、寄送、回寫狀態。
- 寄送資料來源為 `單據紀錄表` 的 `單據類型=繳費單`。
- 繳費單 Email 使用獨立文案與主旨，不沿用收據 Email 文案。
- LINE push 仍不開放，等家長 LINE 綁定與推播文案確認後再接。

原因：
- 繳費單 PDF 產生後，下一個行政步驟就是通知家長繳費；Email 可以先於 LINE push 開放。
- 繳費單與收據的通知目的不同，文案不可混用。

## 2026-06-12 單據 Email 資料來源 fallback

決策：
- 單據 Email 預覽以 `單據紀錄表` 為主，但需支援舊資料 fallback。
- 繳費單 fallback 讀 `學費結算表` 既有繳費單 PDF。
- 收據 Email 不做 `學費結算表` PDF fallback；收據必須來自 `單據紀錄表` 中真正產生的收據 PDF。
- 領據 fallback 讀 `鐘點結算表` 既有領據 PDF 與待寄送狀態。
- 使用 fallback 資料寄送時，需先回補 `單據紀錄表`，再更新寄送狀態。
- fallback 不可一律視為待寄送；若原結算表已是 `已寄送`，需列入已寄送統計並避免重複寄送。
- 預覽與確認寄送必須共用相同狀態判斷；不可出現預覽可勾選、確認送出卻判定沒有待寄送資料的落差。
- 收款資料套用只負責寫入收款方式、收據類別與收款日期；產生收據 PDF 後才建立收據單據紀錄，寄送收據 Email 只能使用該收據 PDF。

原因：
- 專案已經有 1 至 5 月正式資料，部分單據由舊流程產生，尚未同步到 `單據紀錄表`。
- 若只讀新表，行政會看到原結算表有 PDF，但 Email 預覽顯示 0 位，造成誤判。
- fallback 是過渡期必要機制，長期仍以 `單據紀錄表` 作為單據中心。

## 2026-06-12 已產生單據總覽狀態摘要

決策：
- 已產生單據總覽保持只讀，不提供勾選與執行。
- 摘要需呈現 PDF、Email 與 LINE 的完整狀態，不只顯示已寄送數。
- 收據總覽需排除錯誤回補的假收據紀錄，避免再次把繳費單 PDF 當成收據。

原因：
- 行政人員回看舊月份時，最需要快速判斷哪張單據已產生、已寄送、待寄送或失敗。
- 只讀總覽若仍顯示 checkbox，容易讓使用者誤以為可以在總覽直接執行高風險操作。
## 2026-06-12 單據版本管理先定義再開放

決策：
- 先定義作廢、補發、重新產生三種單據版本管理規則，暫不開放直接執行。
- 「已產生單據」維持只讀總覽，加入版本管理規則提示。
- 單據後續狀態以 `單據紀錄表` 為準，避免再次從學費結算表、鐘點結算表混用 PDF 與寄送狀態。

原因：
- 繳費單、收據、領據曾發生資料來源混淆，若在規則未定前開放作廢或重產，可能誤寄錯誤單據。
- 補發、作廢、重新產生會影響家長、講師、收款、記帳與稅務追蹤，必須先保留完整稽核軌跡。
- 已寄送的單據不可靜默覆蓋；更正應保留原單並建立新版。
## 2026-06-12 LINE push 先做準備度檢查

決策：
- LINE push 正式執行前，先在「已產生單據」加入只讀準備度檢查。
- 繳費單與收據的推播對象定義為家長/學生；領據的推播對象定義為講師本人。
- 可推播條件為：有 PDF、有對應 LINE User ID、LINE 狀態為未推播、單據未作廢。

原因：
- LINE push 會直接送到個人聊天室，比 Email 更需要先確認對象來源。
- 家長/學生 LINE ID 與講師 LINE ID 來源不同，若未先分流，容易把收據或領據推給錯的人。
- 先做只讀檢查，可在不打擾家長與講師的情況下確認資料品質。
## 2026-06-12 LINE push 第一版採純文字 PDF 連結

決策：
- LINE push 第一版使用純文字訊息，不使用 Flex/message 卡片。
- 家長收到繳費單與收據 PDF 連結，講師收到領據 PDF 連結。
- 推播前必須預覽、勾選、二次確認；推播後回寫 `單據紀錄表`。

原因：
- 單據寄送重點是正確傳達 PDF 連結，純文字較穩定、可讀、容易除錯。
- Flex 卡片適合後續體驗優化，但初版若先加入版型，會增加測試變數。
- 已發生過單據來源混淆，因此第一版優先確保資料來源與狀態回寫正確。

補充：
- 領據舊資料可能只存在 `鐘點結算表`，因此 LINE push 需支援與 Email 相同的 fallback；執行前回補 `單據紀錄表`，避免狀態繼續分散。
- 作廢流程先開放，因為它是補發與重新產生前的安全閥；執行時只標記作廢狀態並保留原 PDF 與資料列。
- 補發流程已開放，因為補發只重寄同一份 PDF，不改內容、不改單號，風險低於重新產生。
- 重新產生仍先做入口與規則預覽，不直接開放，避免在版本控管尚未完成時誤改正式單據。
## 2026-06-12 重新產生單據正式開放

決策：
- `重新產生單據` 從規則預覽改為正式流程，仍需遵守「預覽 → 勾選 → 二次確認 → 執行」。
- 只允許未 Email 寄送、未 LINE 推播、未作廢且已有 PDF 的繳費單、收據、領據重新產生。
- 執行時必須輸入原因；系統會將舊 PDF 移至垃圾桶，重新產生 PDF，並回寫 `單據紀錄表` 的產生狀態、備註與操作人。
- 已寄送或已 LINE 推播的單據不可直接覆蓋；若內容或金額需更正，必須先作廢原單，再建立新版。

原因：
- 重新產生會改變 PDF 內容與正式單據連結，風險高於補發；必須限制在尚未對外寄出的狀態。
- 保留舊 PDF 連結與重產原因，才能追查行政調整歷程。
